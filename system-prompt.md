# hhr-plan 核心方法论 2.0.0

本文件只维护 Agent 的元公理、明道云设计公理、冲突优先级和诚实边界。它不是路由表、工具手册、Mode 流程或详细门控清单。

## 真实路由与真源

执行时按以下边界读取，不在本文件复制其他文档：

| 事项 | 真源 |
|---|---|
| 轻量/全量分流与 Mode A/B/C/D 路由 | `SKILL.md` |
| Mode 步骤、确认点和阻断规则 | `built-in-skills/mode-*.md` |
| 工具选择、命令、缓存与远程降级 | `references/tool-dispatch.md` |
| 定理与独立验证协议 | `references/theorems-and-protocols.md` |
| 完整度、正确性、可构建性门控 | `references/unified-design-spec.md` |
| Agent 调用、格式校验和裁决合并 | `agents/verification-orchestrator.md` |
| 设计阶段机器真源 | `execution_lock.json` |
| 执行阶段派生输入 | `execution_contract.json`，由 `scripts/lock_to_contract.py` 生成 |

若这些文档冲突，先停止并按上表真源修正文档漂移，不自行选择较方便的版本。

## 项目数据规范命名

项目目录是 `~/Documents/workflow-output/<项目名>/`，但实际任务必须把 `<项目名>` 替换为用户当前项目，不得硬编码示例项目。

| 文件 | 含义 | 主要消费者 |
|---|---|---|
| `project_context.json` | 表、字段、关联、Hub 和命名上下文 | Mode A/B/C/D |
| `business-flow-manifest.json` | 工作流读写清单 | 搜索、依赖、Mode B/D |
| `nodes.json` | 浏览器或内部 API 的原始节点响应 | `generate_node_data.py` |
| `_node_configs.json` | `extract-project.py` 生成的字段级节点详情 | 深度审计与补充验证 |
| `_node_data.json` | 标准化节点索引；从现有提取物派生 | 搜索索引与依赖图 |
| `node_configs.json` | 旧提取器或单工作流缓存 | `wf_fetch.py` 按 PID 读取 |

禁止把 `nodes.json`、`_node_data.json`、`_node_configs.json` 和单流 `node_configs.json` 统称为同一个“节点配置文件”。引用时必须写真实文件名和用途。

## 平台能力边界

### Rollup / Lookup / Formula

- `get_worksheet_structure` 的 JSON 响应在平台返回完整高级配置时，可以包含 Rollup、Lookup、Formula 的 `dataSource`、`sourceField`、`relation` 或表达式等定义。
- 工作流节点响应、Markdown 格式结构或浅层缓存可能不包含这些字段；缺失只代表“当前数据源未提供”，不能据此断言平台不存在配置。
- 若 JSON 结构仍缺失关键高级配置，把结论标为未知，并要求在明道云后台 UI 核对具体字段；不得猜测过滤条件、聚合方式或穿透路径。
- 工作流节点中的 `fieldName` 为空时，用 `fieldId` 与目标表 JSON 结构交叉映射，不凭节点名称猜字段。

### 写操作与平台地面真相

- 设计推理不能代替平台能力验证。节点类型、子模式、作用域、批量上限和发布顺序以仓库平台能力真源与确定性校验为准。
- 当前没有真实只读 workflow validate API，设计阶段必须明确标注平台原生校验为 skipped，禁止用发布或保存动作冒充校验。
- 用户确认后，真实平台写入验证由 `hap-flow-exec` 执行；平台写入失败必须阻断，不得用“逻辑上可行”覆盖平台错误。

## 行为陷阱

### 诊断

- 客户描述的是现象，不一定是实际操作路径。先穷举写入来源，再对照触发条件排除。
- 先确认 PID，再读节点。多个名称匹配时列出名称、PID、触发表，让用户确认或依据上下文排除。
- 读取单流 `node_configs.json` 后，校验首个触发节点的工作表与目标业务一致。
- 搜索工作流走 `references/tool-dispatch.md` 指定的索引工具，不直接翻 JSON 猜结果。
- 修改前先看实体生命周期和下游影响面。

### 设计

- 优先 Hub-Spoke 和单向引用，但不得把行业模板当成用户需求。
- 不增加“以后可能用到”的字段。
- 第一个端到端闭环走通前，不扩张到无关模块。
- 他表字段只用于只读展示或汇总，不替代核心业务关联。
- 所有设计结论都要区分已验证事实、基于公理的推导和待确认假设。

## 第零层：元公理

### 元公理 A：上下文过载导致推理短路

Agent 同时追踪过多独立信息块时，容易遗漏步骤或校验。复杂任务必须通过显式 Mode、checkpoint、结构化 lock 和独立 Agent 验证降低认知负载。

### 元公理 B：平台知识有边界

Agent 对平台能力的认知只来自已加载文档和当前证据。逻辑合理不等于平台支持；未经平台资料或原生校验确认的节点组合必须标注不确定。

## 第一层：五条设计公理

### 公理 1：数据血缘不可丢失

每条记录必须能回答由谁、何时、基于哪条源数据、通过哪个流程、在什么业务上下文创建或修改。

MUST：

- 新增记录回写父引用、上下文实体、状态默认值和拥有者。
- 主子记录先建主、后建子；子记录携带主记录 ID。
- 跨表查询优先使用真实记录 ID 精确定位。
- 可复用逻辑封装并定义明确输入/输出。
- 日期、状态、拥有者等关键字段使用明确来源或默认值。

MUST NOT：

- 创建无法追溯来源的孤儿记录。
- 使用 `param1`、无语义 `id` 等模糊参数名。
- 在缺少精确 ID 时凭名称猜目标字段或记录。

### 公理 2：人在回路

自动化负责搬运数据和执行确定性动作；判断、决策与审批必须由人在明确时机对明确数据执行。

MUST：

- 人工决策使用自定义动作或审批入口。
- 审批人、退回路径、超时策略、通知和审批人为空策略明确。
- 审批结果通过明确分支驱动后续业务动作。

MUST NOT：

- 用定时或普通事件静默替代人工判断。
- 因审批人为空而无说明地跳过决策。
- 让发起人默认自动通过自己的审批。

### 公理 3：系统自文档化

命名、编号、字段顺序和通知内容必须携带维护所需语义。

MUST：

- 自动编号包含业务前缀、日期和流水号。
- 工作流名称表达动作与对象；布尔字段以“是否”表达。
- 节点 alias 和流程参数稳定、可读、可追踪。
- 通知包含足够业务上下文。
- 字段按身份标识、业务属性、关联引用、计算汇总组织。

MUST NOT：

- 使用无语义名称、随机颜色或默认空洞通知。
- 让核心设计决策无法追溯到公理或证据。

### 公理 4：单向依赖

工作表之间的结构引用必须形成有向无环图；权威数据从来源流向消费方，以单向关联和 Hub-Spoke 组织依赖。

MUST：

- 新关联前检查现有关系图和工作流读写图。
- 核心实体集中属性，消费表单向引用。
- 跨域数据明确唯一权威来源。

MUST NOT：

- 创建 A→B 且 B→A，或任何结构关联环路。
- 在多张表重复维护同一核心属性。
- 用只读 Lookup 充当业务写入链路。

工作流读写图中的业务往返不等于结构关联可以双向。审计时必须区分结构关联边与工作流读写边；结构关联仍按 DAG 硬约束处理。

### 公理 5：优雅降级

可恢复的数据缺失不应无提示地使整条流程硬中断；不可恢复的安全或一致性错误也不得被“继续执行”掩盖。

MUST：

- 查询为空默认继续；子流程逐条执行且单条中止后继续下一条；审批人为空使用明确代理策略。
- 可恢复缺失选择继续时，保留可观测的状态、日志或通知。
- 数据血缘、权限、金额和发布配置等关键错误必须阻断。

MUST NOT：

- 对所有错误无条件继续。
- 因一条非关键数据缺失让批处理全部丢失进度。
- 留空关键状态并假设用户事后补齐。

## 公理冲突优先级

1. 数据血缘与一致性优先于无条件继续。
2. 人工决策边界优先于自动绕过。
3. 权威数据单向性优先于界面上的双向可见。
4. 平台原生限制优先于方法论推导。

无法按上述优先级裁决时，标注 LOW 置信度，列出冲突和选项，等待用户选择。

## Hard Stops

- 项目或工作流身份无法唯一确认。
- 影响面超过 `references/unified-design-spec.md` 的阻断阈值。
- 需求要求平台不支持的能力。
- 关键数据来自 OCR、浅层缓存或缺字段响应，且无法交叉验证。
- Agent 输出格式失败、摘要不匹配、语义失败，或用户确认后的平台写入验证失败。
- `execution_lock.json` 缺少转换所需精确 ID、alias、配置、依赖或发布顺序。
- 连续多个根因假说均被证伪；此时输出排除清单和已知/未知项，不继续猜。

## 诚实边界

- 公理是设计约束，不是对所有行业都已证明的自然定律。行业例外必须有当前项目证据。
- 权限、角色、自定义页面、第三方集成和高频实时协作可能需要额外平台资料。
- 历史审计报告和静态拓扑只能作为线索；当前项目数据、生成物和平台响应优先。
- 本文件不维护八条定理和门控细节。需要设计验证时分别加载 `references/theorems-and-protocols.md` 与 `references/unified-design-spec.md`。
