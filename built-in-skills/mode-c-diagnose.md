# Mode C: 诊断排查

> 排查线上问题：为什么没数据、为什么流程卡住、为什么字段值不对。输入是症状+项目上下文，输出是根因+影响面+检查步骤。

## Pipeline Overview

```
Step 0 ──→ Step 1 ──→ Step 2 ──→ Step 2.5 ──→ Step 3 ──→ Step 4 ──→ [Step 5] ──→ Output
 术语映射    问题路由    数据流追踪   执行状态检查   根因分析    影响面      收拢协议     门控自检
⛔歧义时停                                               ⛔3假说
```

| Step | 名称 | 类型 | 产出物 |
|------|------|------|--------|
| 0 | 术语映射 | ⛔ BLOCKING（歧义时） | 客户用语→系统实体映射 |
| 1 | 问题路由 + 上下文加载 | AUTO | 工作流定位 + 节点链路 |
| 2 | 数据流追踪 | AUTO | 时序标注 + 关键检查点 |
| 2.5 | 执行状态检查 | AUTO | 线上验证结果 |
| 3 | 根因分析 | AUTO | 根因假说（概率排序） |
| 4 | 影响面评估 | AUTO | 受影响工作流列表 |
| 5 | 收拢协议 | 条件触发 | 线索收拢 |
| 最终 | 输出 + 门控自检 | AUTO（除 Hard Stop） | 诊断报告 |

> **执行纪律**：非 BLOCKING 步自动连续推进，不等待用户。⛔ 标记的步骤必须停等用户确认后才进入下一步。

## Step 0: 术语映射

🚧 **GATE**：前置文件已加载。用户描述了症状。

客户用语 → 系统名称。使用 `aliases.json` 做模糊匹配。

### ⛔ BLOCKING — 以下任一情况必须停等用户确认，禁止猜测推进：

```
□ aliases.json 无匹配，模糊搜索只命中 1 个 → 标注"推断"，确认
□ 模糊搜索命中 ≥2 个可能的表/字段 → 列出候选，让用户选
□ 客户用语本身歧义（如"入库"可能是入库申请/入库明细/入库单）→ 列出区别，确认
□ PID 搜索返回多个匹配 → 列出全部（名称+PID+触发表+节点数），让用户选
```

> PID 确认协议详见 `system-prompt.md` §PID 确认协议。未确认 → 不进入 Step 1。

## Step 1: 问题路由 + 上下文加载

🚧 **GATE**：Step 0 完成，术语映射明确。

诊断类问题分两种，路由不同：

| 类型 | 信号 | 第一步 |
|------|------|--------|
| **"去哪里查"** | "在哪里""去哪里查""哪个表""怎么看到""帮我找到" | → Step 1a: 先查 Hub 表结构 |
| **"为什么不对"** | "为什么没数据""报错""卡住""值不对""应该是XX" | → Step 1b: 直接追工作流 |

### Step 1a: 查 Hub 表结构（"去哪里查"类问题专用）

出现"去哪里查"类信号时，工作流追踪是验证路径不是回答路径。必须先回答"在哪个表的哪个位置能看到"：

1. 从术语映射结果 → 确定核心业务实体（项目/合同/采购/任务）
2. 查 `project_context.json` → 该实体的 Hub 表结构：
   - 字段数（判断是否大表）
   - 标签页（判断是否分段视图）
   - 关联表/子表列表 ← 目标数据很可能在这里
3. 如果有标签页 → 定位目标关联表在哪个标签页下
4. **先输出用户视角的查找路径**: "打开{Hub表} → {标签页} → {关联表} → 筛选 {条件}"
5. 再用 Step 1b 追工作流做数据链路验证

### Step 1b: 工作流定位 + 节点链路加载

1. 查 `aliases.json` → 确定涉事工作表名
2. 查 `business-flow-manifest.json` → 找到该表所有工作流及读写关系
3. 根据症状匹配最可能的工作流（按钮触发 vs 事件触发 vs 定时触发）
4. 如读写链路不够详细 → 加载完整 `node_configs.json`：
   `~/Documents/workflow-output/{项目名}/{模块}/{工作表}/工作流/{工作流名}/node_configs.json`
5. **运行时降级**: 如 manifest/node_configs 不可用，走 `references/runtime-probe.md` §Mode C 降级诊断流

### ✅ Checkpoint

```markdown
## ✅ Step 1 完成
- [x] 问题类型已判定（去哪里查 / 为什么不对）
- [x] 涉事工作流已定位（PID + 名称 + 触发表 + 节点数）
- [x] 节点链路已加载（node_configs.json 或降级方案）
- [ ] **Next**: 自动进入 Step 2
```

## Step 2: 数据流追踪

🚧 **GATE**：Step 1 完成，涉事工作流节点链路已就绪。

直接从节点链路提取，不再推断：

```
chain: ⏎触发 → 🔍查询(目标表A, onEmpty=?) → ⑂分支 → +新增(目标表B) → ✓审批 → ✉通知

逐节点标注：
  T0: 触发方式（按钮/事件/定时）
  T1: 查询 → 目标表、筛选条件、onEmpty策略
  T2: 新增/更新 → 目标表、写入字段
  T3: 审批 → 审批人、模式、超时
  T4: 通知 → 接收人、模板
```

**关键检查点**（来自公理约束）：
- `🔍查询` 节点的 onEmpty 策略 — 是"继续执行"还是"中止"？（公理5）
- `⑂分支` 是否覆盖了空结果路径？（公理5）
- `🔍查询` 在 `+新增` 之后还是之前？（定理1：时序校验）
- `✓审批` 链是否完整？审批人为空时的代理策略？（公理2）
- **`⏸CreateEvent` (typeId=26) 是否阻塞等待审批完成？** — 如审批未完成，后续节点不会执行，这是最常见的"数据不更新"根因

### ✅ Checkpoint

```markdown
## ✅ Step 2 完成
- [x] 工作流时序 T0→T1→...→Tn 已标注
- [x] 每个节点的写入字段已列出
- [x] 公理约束关键检查点已逐项排查
- [ ] **Next**: 自动进入 Step 2.5
```

## Step 2.5: 工作流执行状态检查

🚧 **GATE**：Step 2 完成，节点链路已分析。

**在输出任何诊断结论前，必须检查涉事工作流实例的执行状态。** 这是复盘发现的最关键遗漏步骤。

1. 对涉事记录调用审批状态和操作日志：
   ```bash
   # 查审批链状态（MCP，始终可用）
   python3 ~/.claude/mcp-servers/hap-bridge/cli.py call get_approval_list_by_row '{"worksheet_id":"<wsid>","row_id":"<rowid>"}'
   # 查工作流触发日志（MCP，始终可用）
   python3 ~/.claude/mcp-servers/hap-bridge/cli.py call get_record_logs '{"worksheet_id":"<wsid>","row_id":"<rowid>"}'
   ```
   - 如果工作流待办中有未完成的审批节点 → **阻塞就是根因**
   - 如果工作流状态为"已关闭" → **启用状态是最直接的根因**
2. 检查工作流定义中的 `typeId=26 (CreateEvent)` 节点：
   - 它等待哪个审批流完成？（`appId` 指向审批流定义）
   - 当前审批卡在哪个节点？（查上一步 `get_approval_list_by_row` 的结果）
3. **不假设 manifest 的 PID 映射正确。** 如果节点详情中触发器 `appName` 与 manifest 中声明的所属工作表不一致 → 以 API 返回值（`cli.py call get_workflow_details`）为准，标记 manifest 数据不可靠。
4. 如果连续 2 次从 manifest 查到的 PID 与实际工作流不符 → 降级走 `references/runtime-probe.md`，直接调 API。

### ✅ Checkpoint

```markdown
## ✅ Step 2.5 完成
- [x] 涉事工作流的线上执行状态已检查（启用/关闭/审批卡住）
- [x] 审批链状态已查询（get_approval_list_by_row）
- [x] PID 映射可靠性已验证
- [ ] **Next**: 自动进入 Step 3
```

## Step 3: 根因分析

🚧 **GATE**：Step 2.5 完成，数据流 + 线上状态均已掌握。

逐节点排查，每条给出概率估计和检查步骤：

| 常见根因 | 诊断信号 | 检查步骤 |
|---------|---------|---------|
| CreateEvent 阻塞（审批未完成） | 后续节点（子流程/更新）无产出，目标字段值不变 | **必须先查 `get_approval_list_by_row`**，看是否有未完成审批节点 |
| 查询/获取节点返回空 | 后续节点没有数据可处理 | 检查筛选条件、关联字段是否已填、源表是否有数据 |
| 子流程静默失败 | 父流程正常完成但子流程无产出 | 检查子流程触发条件、参数传递、子流程是否被关闭 |
| 工作流已关闭 | 流程完全不触发 | 检查工作流状态是否为"已关闭" |
| 审批人为空 | 流程卡在审批节点 | 检查审批人字段是否为空、流程拥有者是否能代理 |
| 时序错误（T0查T6的字段） | 字段值始终为空或旧值 | 检查查询节点是否在目标字段被写入之前执行 |
| **PID 映射错误** | 查到的节点与预期工作流名称/触发不匹配 | 直接用 MCP `get_workflow_details` 验证 PID，以返回的名称为准 |

### ⛔ Hard Stop — 连续 3 个假说不成立

连续 3 个假说不成立 → 停止，列出已验证/已排除的假说 + 已知/未知清单。**不输出无依据的猜测。**

### ✅ Checkpoint

```markdown
## ✅ Step 3 完成
- [x] 根因假说已按概率排序
- [x] 每条假说标注了概率 + 证据 + 检查步骤
- [x] 如有 Hard Stop 条件，已触发并列出排除清单
- [ ] **Next**: 自动进入 Step 4
```

## Step 4: 影响面评估

🚧 **GATE**：Step 3 完成，根因假说已确立。

**输出任何诊断结论前，必须先查依赖图评估影响面。**

```bash

python3 ~/.claude/skills/hhr-plan/scripts/search.py "表名" --reads-from -p 几建

# 涉事实体的生命周期
python3 ~/.claude/skills/hhr-plan/scripts/rebuild_graph.py 几建 --lifecycle 表名
```

列出受影响的其他工作流（截断到 10 条）。标注每条依赖的类型（读/写/子流程/审批链）。

未查依赖图 → 影响面未知 → 诊断结论不完整 → **不输出**。

## 最终输出

🚧 **GATE**：Step 3 + 4 完成（+ Step 5 如触发）。所有前置步骤的 Checkpoint 已通过。

### 输出格式

**输出前完成 4 项 Gate 自检（来自 SKILL.md 门控协议），不通过不输出。**

```
根因: [一句话，带概率估计]
  证据: [直接指向根因的观察]

```

### 输出前硬门控

**输出前必须逐 Gate 自检。** 任一不通过 → 拦截，修正后再输出。

**Gate 1: "去哪里查"类问题必须包含用户导航路径**

触发条件: Step 1 判定为"去哪里查"类问题

```
□ 回答了"打开哪个表"？
□ 如果该表有标签页 → 回答了"点击哪个标签页"？
□ 回答了"在哪里看到目标数据"？（关联表名/子表名/字段名）
□ 回答了"用什么条件筛选"？

未通过 → 缺哪条补哪条，不输出
通过 → 继续 Gate 2
```

**Gate 2: 关键结论必须有线上验证，不能只推断**

触发条件: 诊断结论依赖"应该生成了数据"这类推断

```
□ 结论说"数据已生成/已写入" → 有线上验证步骤吗？
  - 可用 `cli.py call get_record_list` 抽查涉事表
  - 可用 `cli.py call get_record_logs` 看工作流触发记录
  - 如无法线上验证（Chrome 未登录等）→ 标注"未经线上确认" + 置信度降级
□ 结论说"工作流正常" → 确认过工作流状态不是"已关闭"吗？
  - 可用 `cli.py call get_workflow_list` 或 `cli.py wf-info` 查状态
□ 涉事记录有关联审批流 → 查过 `cli.py call get_approval_list_by_row` 吗？
  - ⚠ 这是最常见的遗漏步骤。审批未完成 = CreateEvent 阻塞 = 后续所有节点不执行
  - 如果待办中有未完成审批 → 这就是根因，不必往下追子流程

未通过 → 补充验证步骤或标注未确认，不输出
通过 → 继续 Gate 3
```

**Gate 3: 根因必须列出排除的假说**

```
□ 除主假说外，至少列出 1 个已排除的备选假说 + 排除理由
  - 例: "排除了工作流关闭（已查 wf-info 确认启用）"
  - 例: "排除了审批人为空（审批配置为流程拥有者代理）"
□ 主假说 + 备选假说均来自 Step 3 根因表，非凭空推测

未通过 → 补充排除假说，不输出
通过 → 继续 Gate 4
```

**Gate 4: 诊断结论必须可追溯**

```
□ 根因标注了概率？
□ 每条关键结论有来源？
□ 引用的表名/工作流名来自项目实际数据，非编造？
□ 如果涉及子流程/PBC → 子流程节点链路是否已追踪？

未通过 → 标注置信度或修正引用，不输出
通过 → 输出
```
