# Mingdao APaaS 设计引擎 — 核心方法论

> 从「几建」项目（76工作表含工作流/321工作流/12业务域）反向蒸馏。
> 这不是"他在几建里怎么做"的模式目录，而是一组**设计公理**——给定任何新场景，从公理推导出设计决策。
> 生命周期: `validated` — 6 个项目 Mode D 审计完成。公理2/3=100%，公理1/4/5 领域敏感。

## Quick Reference — 先跑为敬

```bash
# 环境健康检查
SKILL_DIR=~/.claude/skills/hhr-plan
bash $SKILL_DIR/scripts/doctor.sh

# 同步所有派生数据（项目索引、依赖图、搜索索引）
python3 $SKILL_DIR/scripts/auto_sync.py

# 搜索工作流/字段
S=$SKILL_DIR/scripts/search.py
$S "付款" --type workflow -p 几建          # 模糊搜工作流
$S "付款申请" --writes-to -p 几建           # 反查谁写这张表
$S "金额" --field-search -p 几建            # 找哪个表有这字段
$S "合同" --type workflow --all-projects     # 跨项目搜

# 实体生命周期（谁创建、谁更新、谁消费这张表）

# 拉取工作流节点配置
~/.claude/mcp-servers/hap-bridge/cli.py wf-nodes --raw <PID>

# 获取工作表结构
~/.claude/mcp-servers/hap-bridge/cli.py call get_worksheet_structure <sheetId>
```

### 工具能力边界

| 能查到 | 用什么 | 新能力 |
|--------|--------|--------|
| 工作流列表+节点配置 | `hap-bridge wf-nodes --raw <PID>` | — |
| 工作表字段名+类型 | `hap-bridge call get_worksheet_structure` | — |

| 表单自定义按钮 | **Open API**: `GET /v3/app/worksheets/{id}/custom-actions` | 新增 |

>
> **交叉查表**: `wf-nodes --raw` 返回的节点字段 `fieldName` 为 null，但 `fieldId` 完整。用目标工作表的 `get_worksheet_structure` (JSON) 建立 fieldId→name 映射，一口补齐。

## Behavior Traps — 经常踩的坑

以下每一条都在真实排查中反复出现过。不遵守就会多走弯路。

### 诊断陷阱

- **Do not assume the customer's description is the actual operation path.** 客户说的"我改了金额"可能发生在表单直接编辑、子表按钮触发、审批流程副作用、付款计划填写等完全不同的地方。客户描述的是现象，不是根因。
- **Do not guess the workflow — search first.** 工作流名称可能相似但功能完全不同（如「直接新增采购需求」8节点触发表=周报 vs「直接新增」34节点触发表=采购需求）。不能取第一个搜索结果直接读。
- **Do not read node_configs without verifying the PID.** 读取 node_configs.json 后，校验第一个节点的 `appName` 是否与预期业务上下文一致。
- **Do not use grep/rg on JSON files to find workflows.** 一律走 `search.py`（FTS5 索引）。grep 可能漏掉嵌套 JSON 中的字段、触发条件中的工作表引用。
- **Prefer `rebuild_graph.py --lifecycle` over manually tracing workflows one by one.** 一个实体的完整读写链路可能涉及 5-10 个工作流，手工追踪必然遗漏。
- **穷举写入路径再匹配，不要根据客户描述直接定位。** 列出目标字段/记录的所有写入路径（工作流、表单编辑、Rollup/Lookup 联动）→ 逐一分析触发条件 → 对照现象做排除。

### 设计陷阱

- **Prefer Hub-Spoke topology from the start.** 每个新表默认按 Hub 设计——即使现在只有一个表引用它。后期改拓扑的成本是新建时的 10 倍。
- **Do not add "might need later" fields.** 只加当前工作流明确需要的字段。公理5保证新字段对旧流透明。
- **Do not build the third table before the first end-to-end flow works.** 没走通完整闭环之前，不建第三张表。
- **Prefer 单向关联(29) over 他表字段(30) for core business logic.** 他表字段仅用于只读汇总/报表场景。
- **Do not skip the lifecycle check in Mode B.** 修改现有工作流前，必须看 lifecycle——你改的节点可能被 3 个下游工作流依赖。

### 平台陷阱

- **Rollup/Lookup/Formula 字段配置无法通过 API 获取。** 这类字段是"数据联动变更"类 Bug 的高发区——改了A结果B也变了，根因往往在 Rollup 的过滤条件或 Lookup 的穿透路径上。只能通过明道云后台 UI 查看。
- **界面推送(typeId=17): 一个工作流中仅第一个生效。** 后续推送被忽略，不要在同一条路径上放多个推送节点。
- **审批(旧版 typeId=4) 已弃用。** 必须用 typeId=26（发起审批流程）。
- **获取多条→子流程上限 10,000 行。** 超出不报错但只处理前 10,000 条。
- **验证前先跑 verify-platform.py。** 设计方案完成后，先用确定性脚本检查机械错误：`python3 $SKILL_DIR/scripts/verify-platform.py --lock-file execution_lock.json`

## 第零层：元公理（Agent 自身的不可消除约束）

> 以下公理描述的是 **LLM Agent 的固有边界**，而非明道云平台的设计原则。
> 它们解释"为什么某些规则必然成立"——不是因为"犯过错"，而是因为 Agent 的结构性限制无法绕过。

### 元公理 A: 上下文过载 → 推理短路

> Agent 同时追踪超过 3-4 个独立信息块时，推理必然出现步骤遗漏。这不是疲劳问题，是结构性的——超过工作记忆上限时，必然有信息块被挤出，被挤出的那块恰好是校验步骤，就是错误来源。

### 元公理 B: 平台知识边界

> Agent 对明道云平台能力的认知仅限于已加载的文档。逻辑上合理的节点组合可能在平台不支持。任何未经文档对照的节点配置都是猜测。

### PID 确认协议（强制）

1. FTS 搜索返回多个匹配时，**列出全部匹配项**（名称、PID、触发表、节点数），让用户确认或自行对比后再选
2. 工作流名称可能相似但功能完全不同（如「直接新增采购需求」8节点触发表=周报 vs「直接新增」34节点触发表=采购需求），**不能取第一个直接读**
3. 读取 `node_configs.json` 后，**校验第一个节点的 `appName` 是否与预期业务上下文一致**——不一致则说明拿错了，重新查
4. 如果只有一个匹配但触发表与预期不符，用 `--reads-from` 或 `--writes-to` 反向定位正确的工作流
5. 确认 PID 正确后，**告知用户你选了哪个**（名称+触发表+节点数），再继续下游分析

## 第一层：五条设计公理

> 公理是设计决策的根。遇到几建中不存在的场景时，不需要找「最像的模版」——逐一应用这五条公理，它们会收敛出唯一（或高度约束的）设计选择。

---

### 公理 1: 数据血缘不可丢失

> 系统中的每一条记录，必须能回答：谁、在什么时间、由哪个流程、基于哪条源数据、在什么业务上下文中创建。

**这不是偏好，是系统在 200+ 工作表规模下可调试的前提。**

| 约束 | 必须 (MUST) | 禁止 (MUST NOT) |
|------|------------|----------------|
| 新增记录 | 回写父引用：来源记录、上下文实体、状态默认值、拥有者 | 创建"孤儿记录"——无法追溯到来源的数据行 |
| 主子表创建 | 先建主记录，后建子记录（子需要主的记录ID） | 先子后主，或建子时不回写父引用 |
| 流程参数 | 拼音缩写+id/ID，携带语义 | param1, recordId, id 等无意义参数名 |
| 跨表查询 | 「查询工作表」按「记录ID=流程参数[拼音缩写+id]」精确查找；未获取到→继续执行 | 模糊条件查询；未获取到→中止 |
| 封装复用 | 可复用逻辑→封装模板，定义输入/输出参数 | 在多个工作流中复制粘贴相同节点序列 |
| 获取/计算方式 | 获取关联记录→"直接获取"（快照）；数值/日期运算→"直接计算"（缓存） | "每次使用时动态获取/计算" |
| 日期字段 | "系统触发时间"或明确的计算结果 | 留空让用户手动填写 |

**几建证据**: 162个新增节点全部回写父引用。8个工作流先主后子。25+查询节点精确查找。9个封装。45个获取→直接获取，0个动态获取。9个数值/6个日期→直接计算。

**证伪条件**: 在 2+ 个 50 表以下项目中，系统性去掉回写父引用后排查效率未下降，本公理降级为"启发式"。

---

### 公理 2: 人在回路

> 自动化负责搬运数据和更新状态。涉及判断、决策、审批的操作，必须由人在明确的时机、对明确的数据、通过明确的动作触发。

**系统是"操作台"，不是"自动化引擎"。**

| 约束 | 必须 (MUST) | 禁止 (MUST NOT) |
|------|------------|----------------|
| 触发器 | 人工决策→自定义动作(按钮)；数据级联同步→工作表事件 | 用定时触发代替人工判断；用工作表事件自动触发审批 |
| 审批 | 每级一人或签；转审/加签/退回；退回目标=仅发起节点；限时处理+邮件通知=开启；允许审批时上传附件 | 会签；退回任意节点；关闭限时处理或邮件通知 |
| 审批人为空 | 由流程拥有者代理，不阻塞 | 流程中止或跳过审批节点 |
| 自动通过 | 仅"已审批过的审批人" | 发起人自动通过自己的审批 |

**几建证据**: 按钮触发56个 vs 定时触发12个。全部审批或签。审批人为空→流程拥有者代理。

**证伪条件**: 在 2+ 个项目中，定时触发替代人工判断后系统未出现误操作且效率显著提升，本公理降级为"启发式"。

---

### 公理 3: 系统自文档化

> 223张工作表、293个工作流的系统无法靠人脑记忆。命名、编号、颜色必须携带足够信息，让维护者在列表视图中扫一眼就能理解结构和状态。

**"命名即文档，列表即仪表盘"。**

| 约束 | 必须 (MUST) | 禁止 (MUST NOT) |
|------|------------|----------------|
| 自动编号 | 拼音缩写前缀(2-4字母) + YYYYMMDD + 3位流水号 | 纯数字序号；不携带日期信息 |
| 父子表命名 | `子表名（父表名/操作类型）` | 不加括号区分、英文半角括号 |
| 单选颜色 | 冷→暖语义梯度: 蓝(#2196F3)→青(#08C9C9)→绿(#00C345)→黄(#FAD714)→橙(#FF9300)→红(#F52222) | 随机颜色；红色表示"正常" |
| 流程状态 | 平台默认四色（绿=通过/红=否决/灰=中止/蓝=进行中） | 覆盖默认的流程状态颜色 |
| 工作流命名 | 动词+名词（条件）：动作和触发上下文都在名字里 | 只写名词或笼统描述 |
| 通知消息 | 【字段名】内联引用流程字段值 | 默认消息（丢失业务上下文） |
| 字段命名(布尔) | `是否XX` 开头 | 英文命名或模糊缩写 |
| 字段排列 | 身份标识→业务属性→关联引用→计算汇总 | 随机排列；关联引用放在最前面 |
| 代码生成 | 内置编号无法表达时→JS代码块；Input→计算→Output；开启自动重试 | 多工作流重复相同编码逻辑 |
| 打印模板 | 「获取记录打印文件」+ 自定义Word/Excel模板；文件名=记录标题字段 | 系统默认模板 |

**几建证据**: 80个编号全部拼音前缀。16表括号标注父子。13个JS代码块。85个自定义打印模板。

**证伪条件**: 在 2+ 个同等规模项目中，不使用命名规则的系统维护效率未下降，本公理降级为"启发式"。

---

### 公理 4: 单向依赖

> 所有工作表间的引用关系构成有向无环图(DAG)。数据从权威来源流向消费方，从不回流。

**双向依赖→触发循环、权限混乱、数据一致性不可维护。** 不是"几建没有"——是"不能有"。

| 约束 | 必须 (MUST) | 禁止 (MUST NOT) |
|------|------------|----------------|
| 表间引用 | 关联表(类型29)单向引用；Hub-Spoke星型拓扑 | 双向关联；A→B且B→A |
| Hub表 | 核心实体集中属性、被所有相关表单向引用 | 多表重复存储同一实体属性 |
| 他表字段 | 仅用于只读汇总/报表场景 | 核心业务表中作为主要引用手段 |

**几建证据**: 1014个关联中0个双向。关联表:他表字段=761:166=4.6:1。

**证伪条件**: 在 2+ 个项目中，发现 Hub-Spoke 拓扑本身成为瓶颈且双向关联解决了该瓶颈，本公理降级为"启发式"。

---

### 公理 5: 优雅降级

> 系统绝不应该因为一条数据缺失、一个子流程失败、一个审批人为空而硬中断。

**"继续执行"是默认，"中止流程"是例外。**

| 约束 | 必须 (MUST) | 禁止 (MUST NOT) |
|------|------------|----------------|
| 查询/获取节点 | 未获取到数据→"继续执行之后节点" | "未获取到数据时中止流程" |
| 子流程(多条) | 逐条执行，中止时继续下一条 | 并行执行；中止时不再触发后续 |
| 默认值 | 状态、日期、拥有者字段始终设默认值 | 留空期望用户手动填写 |
| 审批人为空 | 由流程拥有者代理 | 流程卡住等待 |

**几建证据**: 324个获取节点统一"继续执行"。121个子流程逐条执行。162个新增节点全设默认值。

**证伪条件**: 在 2+ 个项目中，发现"继续执行"系统性掩盖了应被捕获的错误，本公理的默认行为从"继续执行"改为"有条件中止"。

---

## 公理间的张力与优先级

当公理冲突时，按以下优先级解决：

1. **公理1 (数据血缘) > 公理5 (优雅降级)**: 宁可流程慢，不产生孤儿数据
2. **公理2 (人在回路) > 公理5 (优雅降级)**: 涉及决策时，宁可等人，不用自动化绕过
3. **公理4 (单向依赖) > 公理1 (数据血缘)**: 反向引用会造成循环时，接受单向可见性

---

## 增量构建框架

五条公理就是为增量构建设计的：

| 公理 | 在增量构建中的作用 |
|------|------------------|
| 公理4 单向依赖 | 新表单向引用旧表。旧表不需要知道新表。加新模块不碰旧模块。 |
| 公理1 数据血缘 | 每条记录自带溯源。未来任何流程需追溯时，无需补字段。 |
| 公理3 自文档化 | 命名携带语义。未来加50张表后，扫一眼能定位相关一切。 |
| 公理5 优雅降级 | 查询未获取到→继续。新模块引用旧表字段时，缺字段不炸。 |
| 公理2 人在回路 | 决策不自动化。业务规则变→改工作流节点，不推翻数据模型。 |

### 构建阶段

```
阶段0: 识别核心实体（不问"需要什么字段"，问"业务围绕哪几个东西转"）
  → 项目、合同、客户、任务 ← Hub候选

阶段1: 核心Hub表（每个Hub只建最少字段: 编号+名称+状态+关键关联+溯源字段）

阶段2: 第一个业务流程（端到端走通一个最小闭环）

阶段3-N: 增量扩展（每次新需求: ①查已有表能复用吗 ②新表单向引用Hub ③新字段优先加源头表 ④旧流不受影响）
```

### 四条硬纪律

1. **没走通完整闭环之前，不建第三张表。**
2. **每个表默认按Hub设计**——即使现在只有一个表引用它。
3. **工作流先画泳道，再建表**——用五问推导法，只推导当前模块需要的字段。
4. **永远不加"可能用到"的字段**——只加当前工作流明确需要的。公理5保证新字段对旧流透明。

---

### Hard Stops（触底停止，不进入门控循环）

| 条件 | 动作 |
|------|------|
| Mode C 连续3个假说不成立 | 停止，列出已验证/已排除的假说 + 已知/未知清单 |
| Mode B 影响面 >10 个工作流 | 标注影响范围，确认后再给方案 |
| 公理间张力无法按优先级裁决 | 标注 LOW 置信度，列出两个公理的要求和冲突点，让用户选择 |
| 数据源不可靠（OCR推断等） | 核心决策标注 MEDIUM 置信度 |
| 需求违反公理 MUST NOT | 直接拒绝，说明违反的公理编号 + 约束条目原文（如"公理4: 禁止 A→B 且 B→A"） |
| 用户需求明确需要双向关联引用 | 不直接拒绝。输出 LOW 置信度方案（单向关联 + 他表字段替代），说明"平台上最接近你需求的方案，真正的双向关联做不到"。标注违反的公理4约束。 |
| 平台不支持的功能 | 直接拒绝，说明平台限制 |

## 诚实边界

### 公理的有效范围
- **数据来源**: 五条公理全部来自单项目"几建"，置信度标注的是"在几建中的一致性"
- **未验证的推断**: 公理5"逐条执行而非并行"可能是平台限制而非设计选择
- **可能缺失的公理**: 权限设计、角色设计、自定义页面——几建中几乎没有覆盖

### 公理间的未解张力

| 张力 | 冲突描述 | 当前操作路径 |
|------|---------|------------|
| 公理2(人在回路) vs 公理5(优雅降级) | 用户忘记按按钮时，人等流程→违反5；自动跳过→违反2 | 输出方案时同时给出两个选项：A. 人工按钮+超时自动提醒（偏公理2）；B. 超时自动通过（偏公理5）。标注各自的公理符合度。 |
| 公理3(自文档化) vs 平台显示上限 | 长父表名+子表名超出显示限制 | 主表用 2-4 字母缩写；子表名中括号内容缩短为关键词（如"合同(项)"而非"合同(项目管理/合同审批)"）。凡缩写必须在 aliases.json 中注册。 |
| 公理1(数据血缘) vs 公理5(优雅降级) | 回写父引用失败→违反1；硬中断→违反5 | 默认继续执行+写错误日志节点；数据血缘缺失的记录在审计时标记 LOW 置信度。 |
| 公理4(单向依赖) vs 业务需求(双向引用) | 见 Hard Stops 中"用户需求明确需要双向关联引用"的处理 | 统一走单向+他表字段替代方案，LOW 置信度输出。

## 已知失效场景

以下场景中，当前公理体系可能不足或需要额外补充：

| 场景 | 为什么公理可能不够 | 当前补救措施 |
|------|-------------------|------------|
| 权限/角色设计 | 几建几乎没有覆盖，公理体系对此沉默 | 输出方案时加 LOW 置信度标注，建议参考平台权限文档 |
| 自定义页面 | 同上 | 同上 |
| 第三方 API 集成 | 公理4(单向依赖)在外部系统不可控时难以强制执行 | 标注"集成点的数据一致性需要外部保障" |
| 超过 300 张表的系统 | 公理3(自文档化)的命名规则在超大规模下可能不够 | 需要目录/索引层，但当前公理未覆盖；建议分区命名 |
| 高频实时协作 | 公理2(人在回路)的审批模式可能在秒级协作中成为瓶颈 | 标注"当前公理面向企业审批节奏，实时协作场景需额外设计" |
