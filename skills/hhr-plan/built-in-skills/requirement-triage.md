# 需求筛选

> 公理推导之前强制执行。判断需求是已存在、可做、不值得做还是需要确认。

## 四问

| # | 问 | 判断逻辑 |
|---|-----|---------|
| Q1 | 已存在？ | 查 `aliases.json` → `project_context.json`。如已存在字段/表/流能覆盖 → 指向现成方案 |
| Q2 | 能做吗？ | 对照公理 MUST/MUST NOT。违反公理 MUST NOT → 直接拒绝 |
| Q3 | 多复杂？ | 新建表数+流数+影响范围。低(≤2表,≤3流,无跨模块) / 中(3-5表或4-8流或2模块) / 高(>5表或>8流或≥3模块或新Hub) |
| Q4 | 值不值？ | 有无更简单方式达到同样目的。比如一个字段能解决就不建一张表 |

## 响应分级

| 判定 | 动作 |
|------|------|
| 已存在 | 指向现成方案，说明已有哪个表/流可以覆盖 |
| 直接做 | 低复杂度且不违反公理 → 进入 Mode A/B |
| 确认后做 | 中高复杂度或有影响面 → 标注风险后确认 |
| 不做 | 说明原因 + 替代方案（更简单的方式/已有方案可覆盖） |

## 复杂度估算

- **低**: 新增 ≤2 表, ≤3 流, 单模块
- **中**: 3-5 表 或 4-8 流 或 2 模块
- **高**: >5 表 或 >8 流 或 ≥3 模块 或 新 Hub
