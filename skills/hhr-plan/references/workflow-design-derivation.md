# 工作流设计推导模型

> 从业务需求自动推导：需要几个工作流、什么触发方式、什么节点链、什么复杂度分级。
> 此模型被 SKILL.md 引用，Mode A/B 设计工作流时自动应用。
> 与 `data-model-derivation.md`（表单推导）配合使用：先推导工作流骨架，再由工作流倒逼表单结构。

---

## 核心原理

工作流设计不是「把业务步骤翻译成节点」，而是「识别数据状态转换 → 选择最简节点链」。

```
业务需求 → 每个状态转换 = 一个候选工作流
                 ↓
            触发方式（谁、什么时机）
                 ↓
            数据流签名（读几张表、写几张表）
                 ↓
            复杂度分级 → 匹配节点链模式
                 ↓
            逐节点公理约束 → 时序标注
```

---

## 六步推导法

### 第一步：识别数据状态转换

每个工作流 = 一次数据状态转换。从泳道中提取：

| 问题 | 回答 → 推导结果 |
|------|---------------|
| 什么数据从什么状态变到什么状态？ | → 一个候选工作流 |
| 谁有权触发这次转换？ | → 触发方式候选 |
| 转换前需要读什么上下文？ | → 查询节点数量 |
| 转换后需要通知谁？ | → 通知节点 |
| 转换后需要级联更新什么？ | → 更新/新增节点数量 |

**规则**: 一个泳道步骤如果同时包含"人工判断"和"数据级联"，拆成两个工作流——人工判断用按钮触发，数据级联用工作表事件触发。

---

### 第二步：触发方式决策树

从业务语义出发，按以下顺序判断：

```
涉及人工判断/决策/审批？
  └─ 是 → 自定义动作(按钮)                      [公理2]
  └─ 否 → 数据变更后自动级联？
      └─ 是 → 工作表事件(新增/修改/删除)
      └─ 否 → 被父流程调用的可复用逻辑？
          └─ 是 → 子流程触发
          └─ 否 → 需要周期性执行(提醒/汇总/清理)？
              └─ 是 → 定时触发
              └─ 否 → 外部系统回调？
                  └─ 是 → Webhook触发
                  └─ 否 → 审批结果驱动后续操作？
                      └─ 是 → 审批事件触发
```

**触发分布参考（几建证据）**:

| 触发类型 | 几建占比 | 典型场景 | 公理 |
|---------|---------|---------|------|
| 自定义动作(按钮) | 30.6% | 提交审批/发起流程/手动操作 | 公理2 |
| 审批事件 | 26.8% | 审批通过后级联创建/更新 | 公理2 |
| 工作表事件 | 25.7% | 字段值变更自动同步 | 公理2 |
| 子流程 | 10.4% | 被父流程调用的可复用逻辑 | 公理1 |
| 定时 | 6.6% | 到期提醒/周报生成/数据清理 | — |

**硬约束**:
- MUST: 涉及人工决策 → 按钮触发（公理2）
- MUST NOT: 用定时触发代替人工判断（公理2）
- MUST NOT: 用工作表事件自动触发审批（公理2）

---

### 第三步：确定数据流签名（读→写）

统计工作流的查询/获取节点数（读）和新增/更新节点数（写），归入四种原型：

| 原型 | 签名 | 几建占比 | 特征 | 典型节点链骨架 |
|------|------|---------|------|-------------|
| Mutator | 1→1 | 48.6% | 查一张表、改一张表 | 触发→查询→更新 |
| Signal | 0→0 | 23.0% | 不读不写，纯通知 | 触发→通知 |
| Creator | 0→1 | 11.2% | 不查就创建 | 触发→新增(+回写父引用) |
| Orchestrator | 2+→2+ | 15.0% | 多表读写编排 | 触发→查询A→查询B→分支→新增C→更新D |

**规则**: 签名决定复杂度下限。Orchestrator 至少 Standard 级别（4-6节点）。

---

### 第四步：匹配复杂度分级 + 节点链模式

#### 复杂度分级

| 分级 | 节点数 | 几建占比 | 处理方式 |
|------|-------|---------|---------|
| Simple | 2-3 | 20% | 单线程，不需子流程 |
| Standard | 4-6 | 38% | 允许一层分支，仍为单一工作流 |
| Complex | 7-10 | 22% | 可包含子流程调用 |
| VComplex | 11+ | 20% | 必须拆分：主流程 ≤6节点 + N个子流程 |

**硬纪律**: 超过 10 个节点不拆分 = 违反元公理 A（上下文过载→推理短路）。

#### 五种节点链模式

| 模式 | 签名 | 典型场景 | 详细说明 |
|------|------|---------|---------|
| Pattern A 数据聚合 | Query→Query→Query | 聚合多源上下文后操作 | `patterns/modules/pattern-a-data-interrogation.md` |
| Pattern B 条件迭代 | DataOp→Branch→DataOp→Branch | 多路径决策树 | `patterns/modules/pattern-b-conditional-iteration.md` |
| Pattern C 查询修改 | 1 read → 1 write | 状态转换（最常见） | `patterns/modules/pattern-c-query-modify.md` |
| Pattern D 纯信号 | 0 read → 0 write | 通知/提醒 | `patterns/modules/pattern-d-pure-signal.md` |
| Pattern E 复杂集成 | 2+ reads → 2+ writes | 多表编排 | `patterns/modules/pattern-e-complex-integration.md` |

**匹配规则**: 数据流签名 → 候选模式。Mutator → 优先 Pattern C；Signal → Pattern D；Orchestrator → Pattern E 或 A+B 组合。

---

### 第五步：逐节点公理约束速查

| 节点类型 | 公理1 数据血缘 | 公理2 人在回路 | 公理3 自文档化 | 公理5 优雅降级 |
|---------|--------------|-------------|-------------|-------------|
| 触发 | 参数命名拼音+id | 决策→按钮 | 工作流命名=动词+名词 | — |
| 查询/获取 | 精确查找(记录ID=参数) | — | — | 空→继续执行 |
| 新增记录 | 回写父引用(四问) | — | — | 默认值(状态/日期/拥有者) |
| 更新记录 | — | — | — | — |
| 审批 | — | 或签+退回仅发起+限时+邮件 | — | 审批人空→拥有者代理 |
| 分支 | — | — | — | 覆盖所有可能值 |
| 子流程 | 传参语义化(拼音+id) | — | — | 逐条+中止继续下一条 |
| 通知 | — | — | 【字段名】内联引用 | — |
| 代码块 | — | — | Input→计算→Output | 开启自动重试 |

---

### 第六步：输出时序标注（定理1）

```
T0: 触发([按钮名/事件]) → T1: 查询([目标表]) → T2: 分支([条件]) → T3: 新增([目标表]) → T4: 通知
     写入: [参数列表]      写入: [查询结果字段]                    写入: [新记录字段]
```

**校验规则**:
- Tn 位置的查询节点只能读 T0..T{n-1} 已写入的字段
- 已知错误模式: 在 T0(按钮触发) 后查询"流程状态""审批状态"——这些是后面审批节点才写入的

---

## 拆分决策标准

什么时候拆成多个工作流 vs. 保持一个：

| 信号 | 决策 |
|------|------|
| 节点数 >10 | 必须拆分 |
| 同一逻辑在 ≥2 个工作流复用 | 封装为子流程 |
| 人工判断和数据级联混在一起 | 拆成按钮触发+事件触发两个工作流 |
| 分支后两条路径完全独立（无合并） | 考虑拆成两个工作流，各自触发 |
| 审批通过后有复杂级联 | 审批工作流 + 级联工作流（审批事件触发） |

什么时候合并：

| 信号 | 决策 |
|------|------|
| 两个工作流只差一个分支条件 | 合并为一个，用分支节点区分 |
| 两个工作流操作同一张表的不同字段 | 合并，减少触发次数 |
| 合并后节点数 ≤6 | 可以合并 |
| 合并后节点数 >10 | 不合并 |

## 示例: 从零推导「采购审批」工作流

**需求**: 采购员提交采购申请，主管审批，通过后自动生成采购合同，否决则通知采购员。

**第一步: 识别状态转换**

| 状态转换 | 触发者 | 读上下文 | 通知 | 级联 |
|---------|-------|---------|------|------|
| 采购申请: 草稿→待审批 | 采购员 | 无 | 主管 | 无 |
| 采购申请: 待审批→已通过/已否决 | 主管(审批) | 采购申请详情 | 采购员 | 通过→生成合同 |
| 采购合同: 自动创建 | 审批通过事件 | 采购申请 | 无 | 无 |

→ 3 个候选工作流。

**第二步: 触发方式**

| 工作流 | 触发方式 | 决策依据 |
|-------|---------|---------|
| 提交采购申请 | 按钮 | 人工决策→按钮(公理2) |
| 采购审批 | 按钮(含审批节点) | 人工决策→按钮(公理2) |
| 生成采购合同 | 审批事件 | 审批结果驱动级联 |

考虑合并: 提交+审批可合并（同一张表、操作连贯、合并后 ≤6 节点）→ 合并为一个工作流。
最终: 2 个工作流。

**第三步: 数据流签名**

| 工作流 | 读 | 写 | 原型 |
|-------|----|----|------|
| 提交采购审批 | 0 | 1(更新状态) | Creator→Mutator混合 |
| 生成采购合同 | 1(采购申请) | 1(采购合同) | Mutator |

**第四步: 复杂度+模式**

| 工作流 | 节点估算 | 分级 | 模式 |
|-------|---------|------|------|
| 提交采购审批 | T0触发→T1审批→T2分支→T3通知 = 4 | Standard | Pattern C 变体(含审批) |
| 生成采购合同 | T0触发→T1查询→T2新增→T3通知 = 4 | Standard | Pattern C |

**第五步: 公理约束速查**

- T1 审批: 或签+退回仅发起+限时+邮件+审批人空→拥有者代理 (公理2+5)
- T2 新增采购合同: 回写父引用→采购申请(公理1)，默认值: 状态=草稿、日期=触发时间(公理5)
- T1 查询采购申请: 精确查找(记录ID=cgsqid)，空→继续执行(公理1+5)

**第六步: 时序标注**

```
工作流1: 提交采购审批
  T0: 按钮触发(提交采购审批)     写入: [cgsqid]
  T1: 审批(主管, 或签)           写入: [审批状态, 审批意见]
  T2: 分支(审批状态=通过?)
  T3a: 通知(采购员, 审批通过)     ← 通过路径
  T3b: 通知(采购员, 审批否决)     ← 否决路径

工作流2: 生成采购合同(审批通过触发)
  T0: 审批事件触发(采购申请.审批通过) 写入: [cgsqid]
  T1: 查询(采购申请, 记录ID=cgsqid)  写入: [采购申请全部字段]  ← 读T0的cgsqid ✓
  T2: 新增(采购合同)                 写入: [合同编号, 采购申请(回写), 状态=草稿]
  T3: 通知(采购员, 合同已生成)
```
