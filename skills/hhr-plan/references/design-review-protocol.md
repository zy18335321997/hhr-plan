# Design Review Protocol

> 在每次设计输出前，加载此协议对方案做独立审查。审查者不继承主会话的推理偏差。

---

## 核心原则

> 公理是推导起点，不是事后检验表。
> 不是"方案出完了勾一下公理"——是从公理往下推，推导不出方案的方案就是错的。

---

## Phase 0: 公理推导（必须最先执行）

在给出任何设计之前，逐条走：

```
公理1 (数据血缘) → 这个方案创建的记录，能不能追溯到谁、什么时候、从哪条数据来的？
  ├─ 不能 → 方案不可行。返回修正。
  └─ 能 → 继续。

公理4 (单向依赖) → 数据流是单向的吗？有没有 A→B 又 B→A 的环路？
  ├─ 有环路 → 方案不可行。
  └─ 无环路 → 继续。

公理5 (优雅降级) → 每一步如果数据缺失，流程会硬中断还是会继续？
  ├─ 会中断 → 必须重新设计容错逻辑。
  └─ 会继续 → 继续。

公理2 (人在回路) → 涉及判断/决策的节点是人触发的还是自动触发的？
  ├─ 自动触发且涉及决策 → 违规。
  └─ 人触发或纯数据搬运 → 继续。
```

---

## Phase 1: 逻辑必然性检查

### 1.1 永假命题检测

**问题**: 方案中的判断条件，是否存在"永远不可能为真"的路径？
- 例：全部合同审批状态=通过。如果存在永久性否决/作废的合同，这个条件就是永假的。
- **出现永假条件 → 方案不可行。返回重做。**

### 1.2 逻辑完备性检测

**问题**: 方案的每个分支，都覆盖了所有可能的状态吗？
- 列出目标字段的所有可能值
- 逐一检查每个值会走哪个分支
- 发现未覆盖的状态 → 补充或标记为已知限制

### 1.3 最简性检测

**问题**: 能不能删掉一半节点仍然达到同样效果？
- 如果有两个节点做同一件事 → 合并。
- 如果子流程可以展开为 2 个节点 → 展开。
- 如果方案超过一个工作流但问题声明只需一个 → 质疑假设。

---

## Phase 2: 行为守则检查

### 2.1 迎合检测

**信号**（满足任一即停止）:
- 用户连续纠正 ≥2 次 → 你在执行模式，不在思考模式。
- 方案在用户提出质疑后立即转向 → 你没有自己的判断基线。
- 没有说过"这里有问题"或"更简单的做法是" → 你在配合。

**动作**: 停下，列出你真正不同意的点，说出来。

### 2.2 过度设计检测

**信号**:
- 方案涉及 ≥3 个新建工作流 → 过度。
- 方案涉及新建表 + 新建流 → 先问：现有表能复用吗？
- 节点数 ≥15 → 考虑拆分为多个简单流，或删减。

### 2.3 假设隐藏检测

**问题**: 方案中哪些配置依赖于你未验证的假设？
- 列出每个假设
- 标注置信度：HIGH（代码中确认过）/ MEDIUM（公理推导）/ LOW（猜测）
- LOW 置信度的假设必须在输出中明确标注为 [需确认]

---

## Phase 3: 硬停条件

以下任一条件触发，**不允许输出最终方案**，只输出当前已知/未知：

| 条件 | 动作 |
|------|------|
| Phase 0 任何公理违规且无法修正 | 停止。告知用户冲突点和为什么无法绕开。 |
| Phase 1.1 检测到永假条件 | 停止。说明为什么是永假的。 |
| Phase 1.3 方案超过 1 个流但问题声明只需 1 个 | 停止。给出 1 个流的版本和 N 个流的版本，让用户选。 |
| Phase 2.1 检测到迎合信号 | 停止。列出你没说的话，一次性说完。 |
| 任何配置依赖于 LOW 置信度假设 | 标注，不停止，但必须在输出中醒目显示。 |

---

## 输出格式

通过审查后，每个设计输出必须按此结构：

```
## 推导链 (Phase 0-1)

从公理X出发 → 约束A → 因此必须Y → 所以节点配置为Z。
（每一步追溯到公理编号，不允许跳步）

## 方案 (最简版)

（节点链，每个节点标注其存在的公理理由）

## 假设清单

| # | 假设 | 置信度 | 如果错了会怎样 |
|---|------|--------|---------------|
| 1 | ... | HIGH/MEDIUM/LOW | ... |

## 已知限制

（这里写 Phase 1 发现但未覆盖的边界）
```

---

## 快速自检卡（输出前默念）

1. 这个方案有没有哪个环节是"其实可以不做"的？
2. 我有没有隐藏什么不确定的东西？
3. 如果用户说"不对"，我最担心他质疑哪一点？
4. 这个方案里有没有我自己都觉得不舒服但没说出口的地方？

卡在任一题 → 说出来再交付。
