# Pattern E: 复杂集成（Complex Integration）

> 多表编排：聚合多个数据源，写入多个目标表。跨表协作场景。
> 数据流签名: 2+ reads → 2+ writes | 几建占比: 15.0%

---

## 场景

- 审批通过后同时生成多个下游记录（合同通过→生成入库单+付款计划+任务单）
- 跨模块数据同步（销售合同变更→同步更新项目+库存+财务）
- 复杂业务闭环（采购申请→询价→比价→定标→合同→入库）

## 最小节点链

```
T0: 触发 → T1: 查询(表A) → T2: 查询(表B) → T3: 分支 → T4: 新增(表C) → T5: 更新(表D) → T6: 子流程(表E)
```

节点数: 7-15（Complex/VComplex）。

## 拆分策略（核心）

| 总节点数 | 处理方式 |
|---------|---------|
| 7-10 | 可复用子逻辑 → 封装子流程；其余保持内联 |
| 11+ | **必须拆分**: 主流程 ≤6 节点（触发+查询+分支+子流程调用），复杂操作全部下放到子流程 |

**拆分模板**:

```
主流程(≤6节点):
  T0: 触发 → T1: 查询(核心上下文) → T2: 子流程A(创建记录) → T3: 子流程B(更新状态) → T4: 通知

子流程A(≤6节点):
  T0: 子流程触发(参数: 父记录ID) → T1: 查询(父记录详情) → T2: 新增(目标表, 回写父引用) → T3: 更新(父表状态)

子流程B(≤6节点):
  T0: 子流程触发(参数: 记录ID) → T1: 查询(目标表) → T2: 更新(目标字段)
```

## 公理约束

| 节点 | 约束 |
|------|------|
| 主子创建顺序 | 先主后子：先创建主记录获得ID，再创建子记录回写父引用（定理5） |
| 子流程 | 逐条执行+中止继续下一条（公理5）；参数用拼音+id 语义化（公理1） |
| 每个新增节点 | 回写父引用四问：谁/何时/来源/上下文（公理1） |
| 多表写入 | 检查写入顺序：被引用的表先写（数据依赖） |

## 使用条件

- ≥2 个数据源 + ≥2 个写入目标
- 跨表数据同步或复杂业务闭环
- 目标: 必须使用子流程封装

## 嵌套深度控制

| 深度 | 说明 | 几建实际 |
|------|------|---------|
| 0 层 | 无子流程调用 | 83.6% |
| 1 层 | 主流程→子流程 | 大部分复杂流 |
| 2 层 | 主→子→孙 | 少量极复杂场景 |
| ≥3 层 | 禁止 | platform-verify 会 fail |

## 与其他模式的组合

- 主流程内通常包含 Pattern A（聚合查询）+ Pattern B（条件分支）
- 每个子流程内部通常是 Pattern C（查询-修改）
