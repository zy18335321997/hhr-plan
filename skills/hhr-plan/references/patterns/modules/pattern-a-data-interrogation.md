# Pattern A: 数据聚合查询（Data Interrogation）

> 从多个上下文表聚合信息后，再做操作或判断。查询密集型。
> 节点签名: Query→Query→Query(→操作) | 几建占比: ~15% (complex workflows)

---

## 场景

- 审批前需要聚合多个关联表的背景信息（合同+项目+预算+供应商）
- 生成报表/汇总数据需要跨表采集
- 复杂校验需要比对多个数据源

## 最小节点链

```
T0: 触发 → T1: 查询(表A) → T2: 查询(表B) → T3: 查询(表C) → T4: 操作/分支
```

节点数: 4-7（Standard/Complex）。

## 公理约束

| 节点 | 约束 |
|------|------|
| 每个查询节点 | 精确查找+空→继续执行（公理1+5） |
| 查询顺序 | 先窄后宽：先用记录ID精确查，再用关联字段扩展查 |
| 获取方式 | 直接获取（公理1），不用动态获取 |

## 查询链设计规则

```
1. 起点: 触发记录本身的字段（T0 已有）
2. 第一跳: 用关联表字段获取直接关联的记录
3. 第二跳: 用第一跳结果中的关联字段继续扩展
4. 终止条件: 所有后续操作需要的字段已齐备
```

**硬约束**: 连续查询 >4 个 → 考虑封装为子流程，或检查是否存在冗余查询（两个查询的结果可合并）。

## 使用条件

- 需要 ≥3 个不同数据源的信息
- 查询后操作较简单（一次更新或一次分支）
- 目标: 查询节点占比 ≥50%

## 与其他模式的组合

- Pattern A + Pattern C: 多源查询后执行一次状态转换
- Pattern A + Pattern B: 多源查询后进入条件分支决策
