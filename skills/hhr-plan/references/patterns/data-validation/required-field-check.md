# Required-Field-Check: 必要字段检查

> 在工作流执行关键操作前检查必要字段是否已填。不通过→通知后继续，不中止。
> 节点签名: 触发→分支(字段为空?)→空:通知/非空:继续 | 节点数: 3-4

## 场景

- 提交审批前检查必填字段
- 生成报告前检查数据完整性
- 发送通知前检查收件人字段

## 节点链

```
T0: 按钮触发 → T1: 分支(字段A为空? OR 字段B为空?)
    ├─ 任一为空 → T2a: 通知"请填写【缺失字段名】" → [用户补填后重新触发]
    └─ 全部已填 → T2b: 继续执行
```

节点数: 3-4 (Simple)。

## 检查清单

```
□ 身份标识字段（编号、名称）— 不可为空
□ 业务属性字段（金额、日期、类型）— 有默认值（公理5）
□ 关联引用字段（项目ID、客户ID）— 关联是否存在（ref-existence-check）
□ 审批人字段 — 如为空→流程拥有者代理（公理2+5）
```

## 与平台表单校验的区别

| | 工作流校验 | 平台表单校验 |
|--|----------|------------|
| 触发时机 | 工作流执行时 | 表单提交时 |
| 适用 | 跨字段/跨表条件校验 | 单字段格式校验 |
| 失败行为 | 通知+继续（公理5） | 阻断提交 |
| 优势 | 复杂逻辑+数据链路 | 即时反馈 |

## 公理约束

| 约束 | 规则 |
|------|------|
| 校验失败 | 通知用户（指明缺失字段），不中止（公理5） |
| 默认值 | 状态/日期/拥有者字段必须设默认值（公理5） |
| 字段引用 | 使用项目中实际的字段名（公理4：字段存在性） |
