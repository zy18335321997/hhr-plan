# Retreat-Refill: 退回重填

> 审批否决后退回发起节点，发起人修改后重新提交审批。公理2要求退回目标=仅发起节点。
> 节点签名: 触发→审批→分支(否决?)→退回→重新提交→重新审批

## 场景

- 请假申请被拒→修改日期后重新提交
- 报销单被退回→补填发票信息后重提
- 采购需求被否决→修改数量和规格后重新提交

## 节点链

```
T0: 按钮触发 → T1: 获取审批人 → T2: 审批(或签)
→ T3: 分支(审批结果?)
    ├─ 通过 → T4a: 更新状态=已通过 → T5a: 通知
    └─ 否决 → T4b: 更新状态=已退回 → T5b: 通知发起人（含退回原因）
         → 发起人修改记录 → 重新点击"提交审批"
```

节点数: 6-8 (Standard)。

## 退回规则

| 规则 | 约束 |
|------|------|
| 退回目标 | **仅发起节点**（公理2），非任意节点 |
| 退回原因 | 必须填写，不可为空（公理3：通知含上下文） |
| 重提交 | 发起人修改后重新触发同一工作流 |
| 退回次数 | 不限制，但每次重新计算审批时限 |

## 禁止模式

```
❌ 退回到任意节点 — 违反公理2
❌ 退回后自动重新提交 — 需要人工重新判断（公理2）
❌ 退回时不通知发起人 — 违反公理3（通知携带业务上下文）
```

## 公理约束

| 约束 | 规则 |
|------|------|
| 退回目标 | 仅发起节点（公理2） |
| 退回通知 | 必须包含退回原因+修改建议 |
| 重新提交 | 人工操作，不自动（公理2） |
| 历史记录 | 每次提交/退回/重新审批都写日志 |
