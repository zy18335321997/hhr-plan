# 表单关联关系推导模型

> 从业务需求自动推导：需要哪些工作表、哪些关联表、哪些子表、哪些回写父引用。
> 此模型被 SKILL.md 引用，Mode A/B 设计时自动应用。

---

## 核心原理

表单设计不是「猜到未来会用到什么」，而是「工作流节点需要读/写什么数据」倒逼表单结构。

```
业务场景 → 工作流步骤 → 每步需要读什么？产出什么？
                              ↓
                    读 → 表单必须有对应的 关联表(29) 或 他表字段(30)
                    产出(多条) → 子表(34)
                    产出(单条,被后续引用) → Hub表 + 其他表回写父引用
```

---

## 五问推导法

每条业务需求，依次问这五个问题。每个问题的答案直接生成字段类型。

### 问1: 审批/处理人需要看到什么背景信息？

| 需要看的信息 | 字段类型 | 示例 |
|------------|---------|------|
| 属于哪个项目 | 关联表(29)→实施项目 | 报销单.项目名称 |
| 预算还剩多少 | 他表字段(30) | 报销单.剩余预算 |
| 供应商是谁 | 关联表(29)→供应商信息 | 采购合同.供应商信息 |
| 合同金额多少 | 他表字段(30) | 付款计划.合同金额 |

**规则**: 审批流中每个「获取关联记录」节点的数据源 → 必须在表单上有一个关联表字段。

---

### 问2: 产出什么记录？这些记录后续谁会用？

| 产出类型 | 结构设计 | 示例 |
|---------|---------|------|
| 产出单条主记录，被多个下游流程引用 | Hub表, 集中属性 | 入库申请被出库/库存/财务引用 |
| 产出多条明细，每条独立走流程 | 子表(34), 子记录有独立工作流 | 采购需求→询价明细→每条单独生成询价表 |
| 产出记录只需被一个下游读取 | 关联表(29), 下游回写父引用 | 对公转账→费用明细(对公转账) |

**规则**: 被 ≥3 个其他表引用的 → Hub表。子记录需要独立操作(审批/报价/入库)的 → 子表。子记录只需被查看的 → 关联表。

---

### 问3: 数据溯源链是什么？每条记录必须能回答的四个问题？

每一条新增的记录，必须能回答：

| 问题 | 对应字段 | 字段类型 |
|------|---------|---------|
| 谁创建的？ | 拥有者/创建人 | 成员(26) |
| 什么时候？ | 创建时间/系统触发时间 | 日期时间(16) |
| 从哪条源数据来的？ | 源记录引用 | 关联表(29) |
| 在什么业务上下文中？ | 上下文实体引用 | 关联表(29) |

**规则**: 工作流中每个「新增记录」节点的回写父引用字段 = 上述四个问题的最少字段集合。缺一个就无法追溯。

---

### 问4: 什么逻辑被多处复用？

| 复用次数 | 处理方式 | 需要什么 |
|---------|---------|---------|
| ≥2 个场景用到同一段逻辑 | 封装业务流程 | 输入参数(至少一个记录ID)，对应表单上必须有关联表字段提供这个数据源 |
| ≥3 个节点且单场景使用 | 内联子流程 | 数据源来自获取关联记录(多条) |
| <3 个节点 | 保持内联 | 不单独抽离 |

**规则**: 封装时的输入参数 = 表单上必须有一个字段能提供这个参数值。

---

### 问5: 有没有一对多且每条需要独立操作的数据？

| 场景 | 用子表(34) | 用关联表(29) |
|------|----------|------------|
| 需要逐条审批/报价/入库 | ✅ | ❌ |
| 只需展示/汇总 | ❌ | ✅ |
| 每条有独立的生命周期 | ✅ | ❌ |
| 子记录由工作流自动创建 | ✅ | 子流程逐条新增 |

**规则**: 独立操作(审批/处理/流转) = 子表。只读展示 = 关联表。

## 推导流程（设计时按此顺序执行）

```
输入: 一段业务需求描述
  │
  ├─ Step 1: 画泳道
  │   列出: 谁→做什么→需要什么信息→产出什么
  │
  ├─ Step 2: 标注数据需求
  │   每个步骤标注:
  │   [读] 需要读取的已有数据 → 问1
  │   [写] 需要产出的新数据 → 问2
  │
  ├─ Step 3: 应用五问
  │   问1 → 确定关联表(29)和他表字段(30)
  │   问2 → 确定Hub表和回写父引用方向
  │   问3 → 确定每个新增节点的回写父引用字段清单
  │   问4 → 确定封装/子流程的输入参数和数据源
  │   问5 → 确定子表(34)
  │
  ├─ Step 4: 聚合为表单
  │   合并所有需要的字段 → 去重 → 按字段排列规则排序
  │
  └─ Step 5: 公理校验
      公理4: 检查是否有循环引用
      公理1: 检查每个新增节点是否有完整的回写父引用
      公理3: 检查命名是否符合规范
```

## 示例: 从零推导「进货验收」模块

**需求**: 采购到货后，仓库验收，合格入库，不合格退货。

**Step 1: 泳道**

```
采购员 → 通知到货
仓库 → 验收(需要: 采购合同信息、货品清单) → 填写验收结果
  → 合格 → 生成入库单 → 更新库存
  → 不合格 → 生成退货单 → 通知采购
```

**Step 2-3: 标注+五问**

| 步骤 | 需要读 | 需要写 | 推导的字段 |
|------|--------|--------|-----------|
| 通知到货 | 采购合同(合同号/供应商/货品) | — | 问1→验收单.采购合同(关联表29) |
| 仓库验收 | 采购物品明细(逐条验收) | 验收结果 | 问5→验收单.验收明细(子表34) |
| 合格入库 | 验收通过的数量 | 入库申请 | 问2→入库申请是Hub; 问3→入库明细回写父引用 |
| 不合格退货 | 验收不合格的数量 | 退货单 | 问2→退货单被采购引用; 问3→退货明细回写父引用 |

**Step 4: 表单结构**

```
验收单
├── 自动编号: YSD + YYYYMMDD + 001
├── 采购合同 (关联表29→采购合同)        ← 问1
├── 验收人 (成员26, 默认当前用户)
├── 验收日期 (日期15, 默认当天)
├── 验收明细 (子表34→验收明细)           ← 问5
│   ├── 货品信息 (关联表29→货品信息)
│   ├── 采购数量
│   ├── 实收数量
│   ├── 合格数量
│   ├── 不合格数量
│   └── 备注
├── 验收结论 (单选11: 全部合格/部分合格/全部不合格)
└── 【审批状态】 (单选11: 待验收/已验收)

入库申请                              ← 由验收工作流自动创建
├── 自动编号: RKSQ + YYYYMMDD + 001
├── 验收单 (关联表29→验收单)           ← 问3: 回写父引用
├── 采购合同 (关联表29→采购合同)       ← 问1
├── 入库明细 (子表34→入库明细(入库))    ← 问5
└── ...
```

**Step 5: 公理校验**

- 公理4: 验收单→采购合同(单向) ✓, 入库申请→验收单→采购合同(无循环) ✓
- 公理1: 入库申请回写父引用→验收单 ✓, 入库明细回写父引用→入库申请 ✓
- 公理3: 编号YSD ✓, 父子表命名验收明细(验收单) ✓
