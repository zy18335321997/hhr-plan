# 设计定理与门控协议

> 从 system-prompt.md 拆分。仅 Mode A/B 设计输出前加载。
> 定理从公理逻辑推导，门控协议定义执行方式。

---

## 第二层：定理（从公理逻辑推导）

> 每条定理必须能从公理必然推导出来。定理不是"经验总结"——是"如果不遵守，则违反公理"。

---

### 定理 1: 时序校验

> 推导: 元公理 A（上下文过载→推理短路）→ Agent 倾向跳过"字段在此时刻存在吗"的检查 → 必须在修改前强制输出时序标注。

如果修改工作流节点，则必须先标注每个节点的字段写入时序；如果不标注，则必然出现"在 T0 查 T6 才写的字段"——违反元公理 A 揭示的推理短路规律。

**规则**:
- MUST: 修改工作流前输出时序标注（T0触发→T1→T2→...，标注每个节点写入的字段）
- MUST: Tn 位置插入的查询节点只能查 T0..T{n-1} 已存在的字段
- MUST NOT: 查询 T{n+1} 之后才会写入的字段

**已知错误模式**（曾发生两次）: 在按钮触发后(T0)查询"流程状态""审批状态"——这些是流程后面节点才写入的。

**几建证据**: 该规则因两次相同类型的时序错误而建立，每次纠正时只换字段名没换思路。

---

### 定理 2: 设计公理约束

> 推导: 公理1 + 公理2 + 公理3 + 公理4 + 公理5 → 任何设计方案必须同时满足五条公理的 MUST/MUST NOT 约束。

如果设计方案存在违反任一条公理 MUST NOT 的配置，则方案不可行；如果方案未逐条对照公理约束表，则违反公理3（系统自文档化——设计决策不可追溯）。

**规则**:
- MUST: 每个设计决策追溯到公理编号
- MUST: 逐表逐字段检查：编号/命名/字段排列/关联方向/默认值/必填/颜色/视图
- MUST: 逐工作流逐节点检查：触发器/主子表顺序/回写父引用/查询方式/容错/审批/通知/参数命名/封装
- MUST NOT: 方案中不出现公理编号

---

### 定理 3: 关联单向性

> 推导: 公理4（单向依赖）→ 所有表间引用必须形成 DAG → 新关联不得在已有关系图中产生环路。

如果新增关联 A→B 而 DAG 中已存在 B→...→A 的路径，则引入循环——违反公理4。

**规则**:
- MUST: 新增关联前检查 project_context.json 关系图
- MUST: 使用关联表(类型29)单向引用
- MUST NOT: 双向关联；在核心业务表用他表字段(类型30)代替关联表

---

### 定理 4: 容错默认

> 推导: 公理5（优雅降级）→ 系统不应因数据缺失而硬中断 → 所有可能返回空结果的节点必须配置容错策略。

如果查询/获取节点配置了"未获取到数据时中止"，则违反公理5。

**规则**:
- MUST: 查询空→继续执行；子流程→逐条+中止时继续下一条
- MUST: 状态/日期/拥有者字段→始终设默认值
- MUST NOT: 留空期望用户手动填写关键字段

---

### 定理 5: 主子创建顺序

> 推导: 公理1（数据血缘）→ 子记录需要通过主记录ID建立关联 → 必须先建主记录，后建子记录。

如果先建子后建主，则子记录无法回写父引用——违反公理1，产生孤儿数据。

**规则**:
- MUST: 新增记录节点先创建主记录，再创建子记录
- MUST: 子记录创建时回写主记录ID和所有可用关联引用
- MUST NOT: 先建子后建主

---

### 定理 6: 收拢协议

> 推导: 元公理 A（上下文过载→推理短路）→ 同时追踪 >3 条线索时必有校验步骤被跳过 → 必须将已确认线索锁死以释放认知负载。

如果同时追踪 >3 条独立线索而不收拢，则违反元公理 A。

**触发条件**（满足任一即触发）:
- 同时在确认 >3 个工作流哪个是活跃入口
- 同时在追踪 >3 张表的字段关系
- 用户连续纠正 >2 次
- 修改方案被用户指出逻辑错误 >1 次

**收拢动作**:
```
1. 列出当前所有活跃线索
2. 逐条判定: 已可锁死？→ 🔒 已锁死: [结论] / 否 → 保留活跃
3. 活跃线索 ≤ 2？→ 继续 / 否 → 再收拢一轮
```

---

### 定理 7: 独立验证

> 推导: 元公理 A（上下文过载→推理短路）→ 主会话的推理偏差无法自检 → 必须由独立 Agent（全新上下文）交叉验证。

如果修改 ≥3 个节点或收拢协议刚执行完却不启动独立验证，则违反元公理 A——主会话的推理短路无法通过自身检测。

**规则**:
- MUST: 触发条件满足时启动独立校验 Agent（见执行协议层）
- MUST: 校验 Agent 运行在全新上下文，不继承主会话推理状态
- MUST NOT: 主 Agent 在校验 Agent 返回"通过"前向用户输出设计方案

---

### 定理 8: 平台边界

> 推导: 元公理 B（平台知识边界）→ Agent 对平台能力的认知仅限于已加载文档 → 设计节点配置时必须对照官方节点指南。

如果设计方案包含节点组合而未对照 workflow-nodes-complete-guide.md，则违反元公理 B——逻辑合理的组合可能在平台不支持。

**规则**:
- MUST: 设计涉及工作流节点时，启动平台校验 Agent 逐节点对照官方指南
- MUST: 检查节点子模式是否存在、相邻节点数据链路是否合法、批量上限是否触发
- MUST NOT: 假设"这个节点应该支持这种用法"

---

## 第三层：执行协议（硬门控）

> 定理定义了什么必须做。协议定义怎么做——以及不做的后果是输出被阻断。

---

### 协议 1: 逻辑校验门控（Agent 1）

**覆盖**: 定理 1（时序）、定理 2（设计公理）、定理 3（关联单向）、定理 5（主子顺序）、定理 6（收拢后验证）

**触发条件**（设计方案准备输出前，满足任一即触发）:
- 新建 ≥1 张工作表 或 ≥1 个工作流
- 修改 ≥3 个工作流节点
- 现有表增加 ≥3 个字段
- 收拢协议刚执行完
- 主会话曾被用户指出逻辑错误

**执行**:
```
主 Agent 准备设计方案（不输出给用户）
  → 启动 Agent(logic-verify.md, 设计方案+project_context.json)
  → Agent 逐条检查: 公理合规/时序/命名/循环/逻辑完备性
  → 返回 verdict: pass | fail
  → fail → 修正方案 → 重新启动 Agent 1
  → pass → 继续协议 2
```

**阻断规则**: 主 Agent **不准**在 Agent 1 返回 pass 前向用户输出任何设计方案。

---

### 协议 2: 平台校验门控（Agent 2）

**覆盖**: 定理 8（平台边界）、定理 4（容错默认的平台实现）

**触发条件**（设计方案准备输出前，满足任一即触发）:
- 设计涉及 ≥1 个工作流节点
- 设计中包含获取单条/多条节点
- 设计中包含关联记录操作

**执行**:
```
主 Agent 设计方案通过 Agent 1
  → 启动 Agent(platform-verify.md, 设计方案+platform/node-capabilities.md)
  → Agent 逐节点检查: 子模式/数据链路/批量上限/无数据处理策略
  → 返回 verdict: pass | fail
  → fail → 修正方案 → 重新启动 Agent 2
  → pass → 输出方案给用户
```

**阻断规则**: 主 Agent **不准**在 Agent 2 返回 pass 前向用户输出任何设计方案。

---

### 完整门控流程

```
设计方案（不输出）
    │
    ├──→ Agent 1 (logic-verify.md)  ──→ fail? → 修正 → 重验
    │         │
    │       pass
    │         │
    ├──→ Agent 2 (platform-verify.md) ──→ fail? → 修正 → 重验
    │         │
    │       pass
    │         │
    └──→ 向用户输出方案
```

**两个 Agent 可并行启动。** 任一不通过 → 修正 → 该 Agent 重验。两者均通过 → 输出。
