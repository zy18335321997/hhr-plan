# 工程项目型 APaaS 设计模式

> 来源：几建(同技智能) — 76工作表 / 321工作流 / 12业务域
> **特殊地位：五条设计公理的全部来源项目。这不是"行业模板"，而是公理体系的原始证据。**
> 适用范围：工程公司、项目制管理、EPC总包、专业分包

## 零、几建 vs 行业模板的关系

几建不是第四个行业模板。它是**方法论本身**：

```
几建 (76表/321流)
  │
  ├── 蒸馏出 → 五条设计公理 (system-prompt.md 核心)
  │     ├── 公理1 数据血缘
  │     ├── 公理2 人在回路
  │     ├── 公理3 自文档化
  │     ├── 公理4 单向依赖
  │     └── 公理5 优雅降级
  │
  └── 同时作为 → 工程项目型参考 (本文件)
        └── 与制造业/供应链/政务模式并列，提供领域骨架

莱尔德/尚策/城市运营 → 验证五公理的跨领域有效性
                      同时提供各自的领域骨架
```

**定位**：五公理 = 设计规则（必须遵守），领域参考 = 加速器（建议参考，客户需求优先）。

## 三、几建独有的设计范式

### 3.1 短流程+多关联
```
T0(按钮触发)
  → T1(查询: 关联项目)
  → T2(新增: 任务, 回写项目ID+创建人+状态默认值)
  → T3(通知: 负责人)
```
- 平均5.2节点/流，每流只做一件事
- 复杂业务拆成多条短流程，而非一条长流程
- **这是公理1(数据血缘)和公理2(人在回路)的原始证据**

### 3.2 Hub-Spoke 星型拓扑
```
               ┌──────┐
               │ 项目  │ ← Hub
               └──┬───┘
      ┌─────┬─────┼─────┬─────┐
      ▼     ▼     ▼     ▼     ▼
    合同   任务   成本   质量   安全
```
- 项目是核心Hub，所有业务表通过单向关联指向项目
- **这是公理4(单向依赖)的直接证据：1014个关联中0个双向**

### 3.3 按钮驱动而非事件驱动
- 56个按钮触发 vs 12个定时触发
- 人工决策后才触发的模式：用户点按钮→工作流执行
- **这是公理2(人在回路)的核心证据**

### 3.4 编号体系
- 项目编号: XM-YYYYMMDD-###
- 合同编号: HT-YYYYMMDD-###
- 80个自动编号全部拼音前缀
- **这是公理3(自文档化)的核心证据**

### 3.5 优雅降级的工程实践
- 324个获取节点统一"继续执行"
- 162个新增节点全设默认值
- 121个子流程逐条执行
- **这是公理5(优雅降级)的完整证据链**

## 四、设计原则速查

| # | 原则 | 几建做法 | 对应公理 |
|---|------|---------|---------|
| 1 | 短流程 | 5.2节点/流，每流只做一件事 | 公理1+2 |
| 2 | 项目Hub | 所有业务表单向指向项目 | 公理4 |
| 3 | 按钮驱动 | 人工决策→按钮触发→工作流执行 | 公理2 |
| 4 | 拼音编号 | XM-/HT-/CG- 前缀+日期+流水 | 公理3 |
| 5 | 默认值全覆盖 | 162个新增全设默认值 | 公理5 |
| 6 | 空查询不中断 | 324个获取→继续执行 | 公理5 |
| 7 | 关联表>他表字段 | 1014关联 vs 166他表 = 6:1 | 公理4 |

## 五、快速启动清单（工程项目）

- [ ] 1. 识别核心Hub：通常是"项目"
- [ ] 2. 围绕Hub建业务表：合同/任务/成本/质量/安全，全部单向指向项目
- [ ] 3. 每表最少字段：编号(AutoNumber) + 名称 + 状态(Dropdown) + 关联Hub(Relation) + 备注(Text)
- [ ] 4. 工作流：按钮触发 + 查询(精确查) + 新增/更新 + 通知
- [ ] 5. 每条工作流控制在8节点以内
- [ ] 6. 复杂业务拆分多条短流程，不比一条长流程
- [ ] 7. 编号统一：拼音前缀-YYYYMMDD-###

## 六、与行业模板的使用优先级

> **重要：以下三层是"优先级"而非"互斥"。**

| 层级 | 内容 | 优先级 | 说明 |
|------|------|--------|------|
| L1 设计公理 | 五条公理的 MUST/MUST NOT | **最高** | 不可违反，违反即 Hard Stop |
| L2 领域骨架 | 行业模板的表结构+工作流模式 | **建议** | 提供思考起点，客户需求优先 |
| L3 客户需求 | 客户实际业务流程和个性化需求 | **决定** | 最终设计必须服从客户需求 |
