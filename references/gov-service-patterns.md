# 政务热线/工单系统 APaaS 设计模式

> 来源：城市运营项目反向蒸馏 — 324工作流 / 双核心实体 / 定时报表驱动
> 适用范围：12345热线、随手拍/市民上报、网格化管理、政务服务工单

---

## 一、核心特征：事件汇聚型

城市运营是典型的**事件汇聚型**架构：

| 特征 | 城市运营 | 莱尔德(制造) |
|------|---------|------------|
| 核心实体 | 2个（12345案件+随手拍） | 9域各3-5个Hub |
| 工作流密度 | 66+60条集中在2张表 | 分散到95张表 |
| 定时流 | 45条(14%) | 71条(17%) |
| 审批流 | 2条(1%) | 10条(2%) |
| 命名风格 | 业务对象开头 | 动词+【工作表名】 |

**87%的工作流集中在2张表**——这是"双核驱动"架构，与制造业主的"域隔离"完全不同。

## 三、政务独有模式

### 3.1 热线工单全生命周期
```
市民来电 → 案件录入 → 任务号去重 → 判断重点人员 → 派单 → 处置 → 结案
                                                          ↓
                              超时短信提醒 ← 生成超时日期 ← 满意度测评
                                                          ↓
                                     月报汇总 ← 满意率统计
```

### 3.2 定时报表模式
```
T0(定时触发: 每月1日/每周一)
  → T1(查询: 上月数据)
  → T2(聚合计算: COUNT/SUM/AVG)
  → T3(新增: 月报生成表格)
  → T4(通知: 短信/消息)
```
- 9条月报工作流：热线月报、随手拍月报、人员月报、职能单位月报
- **关键设计**：月报用独立汇总表存储，不实时查询明细（性能考虑）

### 3.3 超时告警模式
```
T0(定时触发: 每小时)
  → T1(查询: 未处置 + 超时时间>阈值)
  → T2(分支: 有结果?)
    ├─ 有 → T3(发送短信)
    └─ 无 → 结束
```
- `热线超期短信提醒`、`生成超时和即将超时日期`
- 政务系统的SLA强制要求（24小时/3天/7天响应）

### 3.4 去重模式
```
T0(新增记录触发)
  → T1(查询: 电话号码+地址匹配已有工单)
  → T2(分支: 是否存在?)
    ├─ 存在 → T3(标记"重复工单"→隐藏)
    └─ 不存在 → T4(正常流转)
```
- `热线12345新增-任务号去重`、`无重复来电时隐藏【重复工单】`

## 四、字段设计特征

### 工单核心字段
| 字段 | 类型 | 说明 |
|------|------|------|
| 工单编号 | AutoNumber | GD-YYYYMMDD-### |
| 来电人/来电号码 | Text | 市民信息 |
| 案件类别 | Dropdown | 多级分类（大类→小类） |
| 案件地址 | Text/关联 | GIS关联 |
| 派单单位 | Dropdown/Relation | 处置单位 |
| 受理时限 | Date+DateTime | SLA计算基准 |
| 处置状态 | Dropdown | 待派→处置中→已结案 |
| 满意度 | Dropdown | 满意/基本满意/不满意 |

### 人员字段（随手拍）
| 字段 | 类型 | 说明 |
|------|------|------|
| 上报人员 | Collaborator | 网格员/市民 |
| 上报位置 | Text/Location | GPS坐标 |
| 上报图片 | Attachment | 现场照片 |
| 督查人/督查结果 | Collaborator/Dropdown | 镇级复查 |
| 考勤情况 | Rollup | 从上报记录汇总 |

## 五、与制造业模式的核心差异

| 维度 | 政务热线 | 制造业 |
|------|---------|--------|
| 实体数量 | 2-5个核心表 | 9域×3-5个Hub表 |
| 数据来源 | 外部(市民来电/上报) | 内部(系统自动/人工录入) |
| 时效要求 | 强制SLA(小时级) | 业务周期(天/周) |
| 报表需求 | 月度/周度/日报 | 实时库存/生产进度 |
| 审批链路 | 几乎不审批(1%) | 审批密集(9%) |
| 外部通知 | 短信/电话回访 | 系统内消息/邮件 |
| 用户角色 | 市民/网格员/处置单位 | 内部员工/供应商/客户 |

## 六、设计原则速查

| # | 原则 | 来源 | 做法 |
|---|------|------|------|
| 1 | 双核驱动 | 87%集中在2表 | 先识别1-3个核心事件实体，围绕它们建工作流 |
| 2 | 定时报表 | 14%定时流 | 月报/周报用独立汇总表，定时批量生成 |
| 3 | SLA告警 | 超时短信提醒 | 定时扫描超时记录→自动通知（短信>消息） |
| 4 | 去重入库 | 任务号去重 | 外部数据录入时先查重，再走正常流转 |
| 5 | 附件驱动 | 随手拍图片 | 现场上报场景必备附件字段 |
| 6 | 级联督查 | 镇级→职能单位 | 多级处置时用子流程或状态流转串联 |
| 7 | 月报独立 | 汇总表≠明细表 | 统计结果存独立表，不实时查询明细 |

## 七、快速启动清单（政务热线项目）

- [ ] 1. 确认核心事件实体（通常2-3个：热线工单/随手拍/信访）
- [ ] 2. 核心表必备：AutoNumber编号 + 来电人信息 + 案件类别(多级Dropdown) + 处置状态 + 受理时限
- [ ] 3. 建处置记录子表（关联主工单）
- [ ] 4. 建月报汇总表（独立于明细表）
- [ ] 5. 工作流：录入→去重→派单→处置→结案→回访(满意度)
- [ ] 6. 定时流：超时扫描+短信通知（每小时），月报生成（每月1日）
- [ ] 7. 随手拍：上报→AI自动分类→镇级督查→整改→验收
- [ ] 8. 移动端适配（随手拍依赖手机拍照+GPS定位）
