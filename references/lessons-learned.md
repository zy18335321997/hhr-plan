## 一、执行经验总结

### 1. 统计归 Python，推理归 LLM

**问题**: Phase 2 最初只用 LLM 读了 15-20 个工作流样本做推理，导致漏掉了大量跨域复现的模式（如记录ID=流程参数、获取关联两种模式）。

**正确做法**: 2209 个节点的频次分布、1014 个关联关系的拓扑分析——这些必须先用 Python 算完（aggregate.py），LLM 只负责在结构化 JSON 报告上做语义推理。

**教训**: 任何涉及「全量数据」的操作（统计、计数、分布、拓扑），写 Python 脚本产出 JSON 报告。LLM 读报告做推理。永远不要让 LLM 自己做统计。

---

### 2. 配置原文 > 统计摘要

**问题**: v1.0 知道「查询节点是第二大节点类型(485个)」，但不知道节点里配了什么。用户一句「查询产品报价单按记录ID=流程参数bjdid」直接暴露了这个盲区。

**正确做法**: 统计数字告诉你「什么多」，只有读配置原文才能发现「怎么做」。高频节点类型发现后，抽样读 ≥10 个配置原文提取子模式（筛选条件写法、容错策略、参数命名）。

**教训**: 每个高频节点类型(>50次出现)必须抽样读原文。发现了子模式后，回到全量数据验证出现次数和跨模块分布。

---

### 3. 术语映射是棕地设计的前置条件

**问题**: 客户说「入库单」，系统里叫「入库申请」。客户说「提交品检」，系统里叫「提交审批」。没有映射表时，引擎匹配到错误的工作流（如32条诊断请求中先匹配到了录用申请而非提交审批）。

**正确做法**: 建立 aliases.json 作为客户语言↔系统语言的桥梁。诊断时先查别名表 → 找不到再模糊搜索 → 还找不到提示用户确认。

**教训**: Mode B/C 启动时，第一件事是术语映射，否则后续所有诊断都可能基于错误的匹配结果。

---

### 4. 先主后子是数据依赖，不是习惯

**问题**: 最初把「先建主记录再建子记录」当作开发者的编码风格。后来发现这是数据依赖：子记录的关联表字段需要主记录的 recordID，主没建之前子无法回写父引用。

**正确做法**: 区分「数据依赖导致的必然顺序」和「个人偏好的可选顺序」。前者置信度 HIGH，后者需要更谨慎的推断。表述为"公理1约束：先建主后建子（子需要父ID）"而非"开发者的习惯"。

**教训**: 每当发现一个「顺序约定」，先检查是否是数据依赖。如果是，标注为 MUST 而非 SHOULD。

---

### 5. OCR 数据源的质量天花板

**问题**: node_configs.json 是从截图 OCR 提取的文本摘要，丢失了原始 JSON 结构。导致节点类型判断依赖关键词匹配、工作表名被 OCR 垃圾文本污染。

**正确做法**: OCR 文本做定性分析（节点类型、大致功能），不做精确字段级提取。设计输出中基于 OCR 文本的节点配置标注置信度 MEDIUM。

**教训**: 优先使用 fields.json（结构化 JSON），降低对 OCR 文本的精度依赖。

---

### 6. 公理骨架 + 项目参数 = 设计引擎

**问题**: v1.x 的 7原则+14启发式本质上是「几建模式目录」，无法应对新场景。用户一问「这是通用的还是针对几建的」就暴露了。

**正确做法**: 从 7 原则提炼为 5 条公理。公理不可变，是引擎的生成能力。启发式可变，是特定项目的实例化证据。每条公理配 MUST/MUST NOT 约束表。

**教训**: 公理 ≤5 条。超过说明还有进一步提炼的空间。

---

### 7. 33条客户需求要有验收单，不是分类报告

**问题**: 第一次分析33条时，按"已实现/未实现"分组排列。用户指出需要升序排列，每条带实现状态和具体操作步骤。

**正确做法**: 逐条编号，每条格式: `#N 标题 + 状态标签 + 现状证据 + 实现步骤`。末尾按实现方式分类汇总。

**教训**: 客户需求分析报告的第一读者是实施工程师，他们需要顺序查找和执行，不是阅读方法论分类。

---

### 8. 工作流解释必须逐节点追踪，不能只给结论

**问题**: 解释"生成询价表"时，用户要求完整的详细说明每一层子流程。

**正确做法**: 16节点主流程 + 11节点子流程 + 11节点孙子流程 = 完整三层节点链。每个节点标注类型、筛选条件、回写父引用。用缩进展示嵌套关系。末尾附设计特征总结。

**教训**: 工作流解释不是给出"大致做了什么"，而是精确到「Node N: 查询工作表 X，筛选 Y=Z，容错策略=继续执行」。

---

### 9. 表单设计不是"猜到未来"，是工作流倒逼

**问题**: 用户困惑"设计表单时怎么能提前想到哪些表需要关联、哪些需要子表"。

**正确做法**: 提炼五问推导法（data-model-derivation.md）——先画泳道 → 标注每步的读写需求 → 五个必问问题 → 聚合为表单字段 → 公理校验。

**教训**: 表单设计和工作流设计不是先后关系，是来回迭代。先画工作流骨架 → 发现缺什么数据 → 补表单字段 → 再精化工作流。

---

### 10. 快照索引和原始数据脱节导致分析不可靠（v2.10.0）

**问题**: 5/29 用 workflow-analyzer 重新提取了原始数据（529 个 node_configs.json 更新），但 build_context.py 没重跑，project_context.json 停留在 5/28。skill 读索引做全局判断、读原始数据做深度追踪，两个版本不一致时结论可能自相矛盾。

**正确做法**: Step 0 加载上下文后立即比对 project_context.json 和最新 node_configs.json 的修改时间。索引落后 → 警告。提取超过 7 天 → 提示。两者同时触发 → 优先修索引，再提示提取。

**教训**:
- 快照不是一个文件，是一个文件链（原始数据 → 索引 → 映射表）。链上任一环节过时，分析结果标注 MEDIUM。
- 时效检查不能靠记忆（"我记得最近更新过"），必须靠文件时间戳比对。
- build_context.py 和 workflow-analyzer 的脱节是结构性风险，方案 B（构建链收口）才是根治，时效检查只是止血。

---

### 11. 诊断"数据在哪里"要先站用户视角，再追工作流（2026-06-15）

**问题**: 客户问「品检通过后的成品入库申请去哪里查」，第一轮诊断直接跳进工作流节点追踪：品检验收表 → 实施项目明细子流程 → 发现 CRUD 写入了入库申请 → 结论告诉客户"去入库申请表查"。客户追问"实施项目表不能查看吗？"才发现遗漏了 Hub 表的标签页聚合视图——实施项目的「采购信息」标签页下直接挂着入库申请，这才是用户要找的地方。

**正确做法**:

```
用户问"XX数据在哪里"时的诊断顺序:
1. 先查业务主表(Hub)结构 — get_worksheet_structure / project_context
   → 看有没有标签页(tab)、关联表(relation)字段
2. 再查此表与目标表的关系 — manifest.read_index/write_index
3. 最后追工作流验证数据确实会写入
```

**教训**:
- **Hub表+标签页是用户的第一导航界面**。用户不会去工作流日志里找数据，她打开主表切标签页。第一回答必须回答"在哪个表的哪个位置能看到"。
- 工作流视角是验证路径，不是回答路径。先给位置，再给链路。
- 127 字段的大 Hub 表（如实施项目）的标签页结构要先读结构再推断，不能靠 manifest 猜。

## 二、用户偏好与理念提炼

### 设计理念

1. **节点级精度优先** — 不接受抽象的设计原则，要求输出能直接在明道云里配置的节点级方案。每个节点必须说明：类型、筛选条件怎么写、参数怎么命名、容错策略选什么。

2. **可操作优先** — 输出必须是「用户可以照着做的操作指南」而非「技术参数说明」。字段要翻译成"填你的名字"而非"成员类型必填"。术语要翻译为客户语言。

3. **真实系统优先** — 不能编造不存在的表名、字段名、工作流名。所有引用必须来自 project_context.json 中的实际工作表/工作流。Mode B/C 下命名空间必须校准到项目实际参数。

4. **诚实标注优先** — 不确定的事情必须标注置信度（HIGH/MEDIUM/LOW）。不知道的就说不知道（诚实边界）。公理间有张力时标注为 LOW 并说明为什么。

5. **系统性思维** — 不能割裂看问题。设计新模块要检查命名冲突、循环引用、与已有工作流的交互影响。诊断问题要同时分析根因和影响面。

6. **一线到底** — 希望一个 `/hhr-plan` 命令走完所有步骤（Step 0 项目选择 → 模式判断 → 推导/诊断），不希望中间停下来问"要不要继续"。

7. **实战验证** — 不接受演示，要求拿真实数据跑一遍看效果。用具体的反例来测试方法论的有效性。

### 命名偏好

| 偏好 | 说明 |
|------|------|
| 中文术语 > 英文术语 | "回写父引用"而非"backfill" |
| 动词+名词 | 工作流命名: "提交审批"而非"审批提交流程" |
| 拼音缩写+id | 流程参数: `cgxqid`, `rksqid` 而非 `param1` |
| 术语一致性 | 一个概念在系统中只有一个名字 |
| 括号标注父子 | `入库明细（入库）`, `费用明细（个人报销）` |

### 交付物偏好

| 偏好 | 说明 |
|------|------|
| 升序排列 | 需求列表按编号顺序，不按类型分组 |
| 逐条验收单 | 每条需求: 状态标签 + 现状 + 实现步骤 |
| 数据流图 | 用缩进+箭头展示工作流节点链 |
| 追溯表 | 每个设计决策追溯到公理编号或"问N" |
| 分类汇总表 | 文档末尾附按实现方式分类的汇总 |

## 三、可复用规则清单（15条 → 更新为20条）

### A. 脚本执行规则

1. **Python 做统计**: 全量数据的统计、分布、拓扑分析，用 Python 脚本产出 JSON 报告。LLM 不循环读文件做统计。
2. **高频模式抽样**: 统计报告中的 top-N 高频模式，抽样读 ≥10 个配置原文提取子模式。
3. **结构化优先**: fields.json（结构化JSON）的数据是可信的。OCR 文本的数据标注置信度 MEDIUM。

### B. 设计输出规则

4. **公理追溯**: 每个设计决策追溯到具体的公理编号。
5. **五问推导**: Mode A/B 必须按五问法推导表单关联关系，不靠"猜"。
6. **置信度标注**: HIGH=数据直接支撑, MEDIUM=公理推导+项目上下文, LOW=公理间张力或纯推断。
7. **口语检验**: 客户可读输出必须默念一遍，读不顺就改。配置术语→客户语言对照表必须遵守。
8. **引用实际名称**: Mode B/C 中的表名/字段名/工作流名必须来自 project_context.json，不编造。
9. **影响面+根因**: 诊断输出 = 术语映射 + 工作流追踪 + 数据流追踪 + 根因分析(+概率) + 影响面分析。

### C. 项目上下文规则

10. **Step 0 必执行**: 每次调用首先检查项目上下文。Mode B/C 无项目 → 列出可用项目或引导新建。
11. **aliases.json 先查**: 客户术语先查别名映射表 → 找不到模糊搜索 → 还找不到提示确认。
12. **新建项目三步**: 收集信息 → build_context.py → 注册 projects_registry.json → 创建 aliases.json 模板。

### D. 质量门禁规则

13. **命名冲突检查**: 新表名在 project_context.worksheets 中不存在。
14. **循环依赖检查**: 新关联不在 project_context.relation_graph 的环路路径上。
15. **命名风格一致**: 新命名符合项目的编号格式、括号标注习惯、工作流动词模式。

### E. 交付格式规则

16. **需求验收升序排列**: 逐条从 #1 到 #N，每条带状态标签（已实现/未实现）+ 现状 + 实现步骤。
17. **工作流逐节点解释**: 不能只给结论，必须逐节点标注类型、筛选条件、容错策略、参数传递。多层嵌套用缩进展示。
18. **设计决策追溯**: 每个设计方案末尾附追溯表，决策→公理编号/问N。
19. **先工作流后表单**: 设计顺序 = 画泳道 → 标注每步读写需求 → 五问推导 → 聚合表单字段 → 公理校验。不在不清楚工作流的情况下设计表单。
20. **对话结束即更新**: 每次对话产生了新经验/新规则/新发现，更新此文档对应部分。
21. **诊断先站用户视角**: 用户问"XX数据在哪里查"时，先查 Hub 表的标签页/关联表结构回答"在哪个表的哪个位置能看到"，再追工作流做链路验证。不倒置。见 Lesson 11。

## 四、自迭代触发条件（让 Rule #20 不再是一纸空文）

> 以下任一条件满足时，必须在对话结束前更新此文档。

### 触发事件

| # | 事件 | 更新位置 | 示例 |
|---|------|---------|------|
| T1 | 新增 hard rule 或修改已有规则 | 对应章节（A-E）+ bump version | 快照时效检查 → 新增 rule 21 |
| T2 | 发现新的错误模式，且连续发生 ≥2 次 | 一、新增教训条目 | 流水号回滚更新 → lesson 8 |
| T3 | 用户纠正 skill 的输出 ≥2 次（同一会话） | 一、新增教训条目 | 时序错误纠正 → hard rule 6 |
| T4 | 发现公理覆盖不到的场景 | 公理约束表 + 诚实边界 | 多项目验证发现新公理 |
| T5 | 发现引用文件过期或矛盾 | 一、新增教训条目 | project_context.json vs node_configs.json 版本不一致 |
| T6 | 新增可复用规则（脚本/设计/格式） | 三、追加到对应分组 | 新脚本规则 → A 组 |
| T7 | 版本号 bump | 版本历史（见下） | Minor: 新规则/教训; Patch: 修正/澄清 |

### 执行约定

```
分析/设计结束 → 检查 T1-T7
  ├─ 无触发 → 结束
  └─ 有触发 → 更新 lessons-learned.md
       ├─ 新教训 → 一、新增条目（问题 + 正确做法 + 教训）
       ├─ 新规则 → 三、追加到对应分组（A-E）
       ├─ bump version → 五、版本历史
       └─ 更新 SKILL.md 的 version 和 references（如有新增引用文件）
```

### 停止条件
- 不能为了"显得学到了"而编造教训。只有 T1-T7 确实触发时才更新。
- 每条新教训必须有具体发生日期和触发上下文，不能是抽象总结。
