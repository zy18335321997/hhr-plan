# Pattern B: 条件迭代（Conditional Iteration）

> 基于数据条件分支处理不同逻辑路径。决策树模式。
> 节点签名: DataOp→Branch→DataOp→Branch | 几建占比: ~12%

---

## 场景

- 审批通过/否决走不同处理路径
- 根据金额/类型/状态分级处理（小额直接通过，大额需审批）
- 库存充足→出库，库存不足→提示补货
- 任何"根据条件走不同路"的多路径逻辑

## 最小节点链

```
T0: 触发 → T1: 查询/操作 → T2: 分支(条件) → T3a: 路径A / T3b: 路径B
```

节点数: 4-8（Standard/Complex）。

## 常见变体

| 变体 | 节点链 | 节点数 | 分级 |
|------|-------|-------|------|
| 二分支 | 触发→查询→分支→路径A/路径B | 4-5 | Standard |
| 多级分支 | 触发→查询→分支→分支→路径A/B/C | 6-8 | Complex |
| 分支后合并 | 触发→分支→路径A/B→合并→通知 | 5-6 | Standard |

## 公理约束

| 节点 | 约束 |
|------|------|
| 分支条件 | 必须覆盖目标字段的所有可能值（逻辑完备性） |
| 每条路径 | 独立执行容错：空→继续执行（公理5） |
| 分支条件 | 不得出现永假分支（logic-verify Phase 1.1 会检查） |

## 分支设计规则

```
1. 列出条件字段的所有可能值
2. 为每个值指定分支路径
3. 检查: 有没有遗漏的值？→ 补充 "其他" 分支或确认不可能出现
4. 检查: 有没有永假分支？（条件永远不成立）→ 删除或标注
```

**硬约束**: 分支层数 >2 层嵌套 → 考虑拆分为多个工作流，每个工作流处理一个分支路径。

## 使用条件

- 业务逻辑有 ≥2 种处理路径
- 不同状态/金额/类型需要不同处理
- 目标: 分支节点占比 10-15%

## 升级路径

- 分支路径内操作 >3 个节点 → 封装为子流程
- 分支后需要操作不同的表 → 考虑 Pattern E（复杂集成）
