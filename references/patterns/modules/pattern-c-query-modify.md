# Pattern C: 查询-修改（Query-Modify）

> 最常见的工作流模式。查一张表、改一张表（或同一张表），完成一次状态转换。
> 数据流签名: 1 read → 1 write | 几建占比: 48.6%

---

## 场景

- 提交审批（查当前记录→更新状态为待审批）
- 审批通过后更新状态（查审批结果→更新业务状态）
- 确认收货（查出库记录→更新库存数量）
- 任何"读一条→改一条"的操作

## 最小节点链

```
T0: 触发([按钮/事件]) → T1: 查询([目标表]) → T2: 更新/新增([目标表])
```

节点数: 2-3（Simple）。加审批或通知后升至 4-6（Standard）。

## 常见变体

| 变体 | 节点链 | 节点数 | 分级 |
|------|-------|-------|------|
| 基础型 | 触发→查询→更新 | 3 | Simple |
| 审批型 | 触发→查询→审批→分支→更新+通知 | 5-6 | Standard |
| 创建型 | 触发→查询→新增(子记录)→更新(父记录) | 4 | Standard |
| 通知型 | 触发→查询→更新→通知 | 4 | Standard |

## 公理约束

| 节点 | 约束 |
|------|------|
| T1 查询 | 精确查找: 记录ID=流程参数[拼音+id]（公理1）；空→继续执行（公理5） |
| T2 新增 | 回写父引用+默认值（公理1+5）；先主后子（定理5） |
| T2 更新 | 仅更新本次状态转换涉及的字段 |
| 审批 | 或签+退回仅发起+限时+邮件+空→拥有者代理（公理2+5） |

## 使用条件

- 单一状态转换，无复杂分支逻辑
- 涉及 ≤2 张表
- 目标节点数 ≤6

## 升级路径

- 节点数 >6 → 考虑 Pattern B（条件迭代）或 Pattern E（复杂集成）
- 同一逻辑 ≥2 处复用 → 封装为子流程
