# Dedup-Check: 去重校验

> 在工作流中检测重复记录，防止数据冗余。几建最常用的校验模式。
> 节点签名: 触发→查询(精确条件)→分支(有结果?)→有:通知+结束/无:新增

## 场景

- 采购需求去重（同一项目+同一物料→不重复创建）
- 客户名称去重（新增前检查是否已存在）
- 任务分配去重（同一人+同一任务→不重复分配）
- 任何"不允许重复"的业务约束

## 最小节点链

```
T0: 按钮触发 → T1: 查询工作表(精确查找)
    筛选: 名称=触发记录.名称 AND 项目=触发记录.项目
    空策略: 继续执行
→ T2: 分支(查询结果>0?)
    ├─ 是 → T3a: 通知"已存在" → [结束]
    └─ 否 → T3b: 新增记录
```

节点数: 4-5 (Simple/Standard)。

## 查询条件设计

```
规则: 精确查找，禁止模糊匹配
好的条件: 名称="XX" AND 项目ID="YYY"
坏的条件: 名称包含"XX"（模糊匹配）
```

## 变体

| 变体 | 节点链 | 场景 |
|------|-------|------|
| 简单去重 | 查询→分支→通知/新增 | 单条件 |
| 多条件去重 | 查询(AND多条件)→分支→通知/新增 | 项目+物料+规格 |
| 去重+更新 | 查询→分支→有:更新/无:新增 | 存在就更新 |

## 公理约束

| 约束 | 规则 |
|------|------|
| 查询方式 | 精确查找（记录ID/名称=），不模糊（公理1） |
| 空策略 | 查询空→继续执行（公理5） |
| 通知 | 发现重复→通知用户（公理3），不静默跳过 |
| 日志 | 去重动作写日志/标记（公理1：数据血缘） |
